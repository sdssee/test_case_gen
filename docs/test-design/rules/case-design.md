# 测试计划与用例规则

## 先计划再写正文

从 `facts.json` 识别独立功能。每个功能先建立一个基线用例意图，再按适用 DFX 场景扩展，使用稳定 `case_id` 写入 `case-plan.json`。计划阶段完成数量与覆盖思考，正文阶段不临时发明功能或事实。

一个用例只能归属一个功能。一个搜索用例中使用重置，不影响重置功能拥有自己的基线与适用 DFX 用例。无需手工维护“主元素/辅助元素”字段，功能归属和事实引用已经表达该关系。

## 正文质量

- 标题为“功能点-当前用例标题”，不得为空。
- 前置、步骤、预期使用列表存储；导出时按 `1. ...` 编号换行。
- 每个步骤对应一个可判定预期；不同用例不得整段复制相同步骤和预期。
- 步骤使用测试人员能看到的菜单、文案、控件和数据，不写内部 ID、选择器、事实编号或页面工具术语。
- 预期必须来自真实观察；禁止“操作正常”“结果符合预期”等空泛语句，也禁止要求截图作为测试动作。
- 同一功能的全部用例连续排列，便于管理。
- 每条用例引用至少一个有效 `fact_id`，审查必须双向验证事实覆盖和用例依据。

分页中 10/20/30 条可以根据测试意图形成一条连续用例，也可以在产品行为显著不同且各自需要独立判定时拆分；不允许只改标题而复制相同正文。
