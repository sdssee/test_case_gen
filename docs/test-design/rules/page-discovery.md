# 页面实探与元素覆盖规则

## 页面材料处理

涉及页面、截图、原型、页面 URL、菜单路径、浏览器或 computer use 时，必须先识别页面元素，再生成或补充用例。

材料优先级：

1. 需求、用户故事、验收标准和业务规则。
2. 用户提供的文档、截图、原型图、操作说明。
3. 可访问页面的实际 DOM、可见文案、菜单、按钮、链接、图标按钮、下拉项、开关、输入框、弹窗和表格行操作。
4. 浏览器或 computer use 观察到的页面行为。
5. 无法确认的信息登记为待确认问题。

## 深遍历要求

- 最终多 Agent 架构中，Discovery Agent 是当前批次唯一实探 owner；不得并行拆分页面后拼接 inventory、discovery、逐选项账本或 CRUD 生命周期。它只能写自己的隔离 workspace，由编排器校验并原子提升到正式账本。

- 当前批次正式写测试用例前，如存在可访问页面、原型或桌面窗口，必须使用浏览器或 computer use 遍历当前批次所有可点击/可交互功能点。
- 执行交互前，必须从 DOM、可访问性树、浏览器 trace 或桌面控件树独立采集 `page-element-inventory.csv`；清单中的可交互元素必须与 `page-discovery.csv` 双向一一对应。不得用已经写好的实探账本反推清单，也不得删掉未实探元素来通过覆盖率。
- 同一元素的不同角色、状态、分支或动作使用不同且稳定的 `交互实例ID`；该 ID 必须在 `page-discovery.csv`、`selection-option-observations.csv`、`interaction-branch-observations.csv` 和 `element-case-plan.csv` 精确传递，防止同名控件或同一控件的多次交互互相覆盖。
- 每个可交互元素都必须实际执行，不得用“已看到控件”“已展开”“理论可操作”代替。`page-discovery.csv` 每行必须记录操作前数据/页面状态、`完整点击路径` 中的具体动作、操作后的真实变化与结果分支、必要的恢复结果、存在的 `证据路径` 和唯一 `证据定位`。证据必须是当前 run-dir 的 `artifacts/` 内非空文件，目录、外部路径或空文件无效。截图证据只能证明该截图捕获的一次交互状态；内容相同的静态截图即使改名、复制或换路径也不得证明多个不同交互。视频、DOM 或 trace 可以复用文件，但定位步骤/时间点必须不同。
- PNG/JPG/PDF/视频/Office 等二进制证据在写入 `artifacts/` 前必须裁剪或遮蔽地址栏、真实 URL/IP、主机、账号和凭据，并在证据旁生成 `<原文件名>.sensitive-audit.json`。sidecar 必须使用 `schema_version=1.0.0`，记录与当前证据一致的 `evidence_sha256`、`inspection_method`（`model_visual_review`/`ocr_and_visual_review`/`manual_visual_review`）、脱敏后的完整 `visible_text`、三个值为 `true` 的 `address_bar_cropped_or_masked`/`environment_identifiers_masked`/`credentials_masked`、`status=PASSED` 和非空 `notes`。缺失、哈希失配、声明不完整或可见文本含敏感值时保持 discovery；sidecar 不能替代实际交互证据。
- 输入类、动态选择、分页和弹窗必须另写 `interaction-branch-observations.csv`：输入覆盖正常/空值/非法/边界，动态选择覆盖有无结果搜索、滚动或分页加载、首中末/边界选择和清空恢复，分页覆盖条数、前后页、跳转、边界、末页/无数据和筛选重置，弹窗覆盖打开、确认、取消、关闭/Esc 和恢复。每个分支必须独立实际执行、独立取证、记录恢复结果并在 plan 阶段绑定一个不复用的用例 ID；禁止在单行堆砌关键词冒充多分支执行。
- 每行 `操作步骤锚点` 必须取自已执行的元素、动作或路径，并进入该计划 owner 的每条关联用例步骤，防止生成无关操作；`预期结果锚点` 必须取自真实观察结果，并至少进入一条精确关联用例的预期结果。异常、边界等变体用例保留各自可判定结果，不得为通过门禁复制正向成功结论。
- `未执行`、`未实际点击`、`仅展开`、`数据不足`、`缺少测试数据`、`无法充分验证`、`待补充` 或证据与账本结论矛盾时，元素不得标为已覆盖，批次必须保持 `DISCOVERY_REQUIRED`。用户确认只能解释页面外的业务语义，不能把未执行功能改成已覆盖，也不能放行 cases/delivery。
- 所有涉及页面实探、截图、原型或可访问系统的任务，默认直接全量深度探索，不需要向用户二次确认是否深探、是否展开下拉、是否尝试新增测试数据；必须捕捉全部可交互元素。只有需要真实账号/验证码/权限开通、疑似生产环境、缺少必要测试数据或涉及真实密钥时才暂停确认；暂停不等于覆盖完成，补齐安全测试条件后必须继续实探。
- 新增/创建必须实际成功；对本次创建且带测试标识的数据，编辑/修改页中的每个可修改元素都必须至少完成一次有效变更，保存后重新进入列表、详情或编辑页检查持久化回显，并通过依赖功能验证实际生效。不得只观察字段、只触发校验或只看到“成功”提示。
- 风险确认只用于全量深探后模型仍无法理解的业务语义、规则歧义、权限含义或页面无法观察项，不得用于询问用户是否继续常规深探。能通过点击、选择、输入、切换、翻页或观察页面状态验证的问题必须由模型完成验证，不得转问用户。
- 具体写测试用例时，必须在对应页面或功能点内做深遍历，覆盖所有可点击、可输入、可测试元素。
- 页面所有按钮、链接、菜单、Tab、图标按钮、表格行操作、更多操作、批量操作、筛选、排序、分页、导入、导出、上传、下载、预览、弹窗、抽屉、确认框、开关、禁用态、空状态、错误态都必须记录和处理。
- 页面元素覆盖清单中的每个元素必须关联功能用例 ID、标记不适用、标记不测范围或登记待确认问题。
- `page-discovery.csv` 和页面元素覆盖清单中的每个可交互元素必须填写适用 `DFX维度` 与 `DFX场景`；已生成用例的元素不得缺失 DFX 映射。
- 页面深探完成后必须生成或更新 `element-case-plan.csv`，把每个页面元素、交互路径、数据状态、适用 DFX、应生成用例数、计划用例ID和实际用例ID建立映射；功能测试用例必须从该计划派生。
- 元素计划必须用 `最小标题路径 + 交互实例ID + 页面/入口 + 元素名称/文案 + 元素类型` 精确匹配，并填写结构化 `操作类别`、`验证要求`、`数据策略`、`执行状态`；禁止仅靠元素名称模糊匹配。
- 编辑、配置和状态变更页面中的每个可修改元素必须在 `test-data-lifecycle.csv` 按修改项独立记录修改前值、修改后值、保存后回显和实际生效结果。

## 选择类控件

- 下拉框、级联选择、单选框、复选框、树选择、枚举筛选等选择类控件不得只展开查看选项。
- 有限选项集合的每个可选项必须逐项实际选择，不得抽样；每个选项都要观察后续字段、二级/三级下拉、列表数据、按钮禁用态、校验提示、默认值、提交前提示和结果分支变化。页面实际禁用、置灰或无权限的选项必须执行选择尝试并记录具体阻塞状态、`是否实际选择=否` 和独立证据；“未知/待确认/可能无权限”不能作为例外。复选框逐个验证原子选项，组合关系再按代表性组合补充；树选择逐个验证可用节点及父子联动。
- 动态、远程搜索或滚动加载选项集合必须逐行记录实际选择过的选项，并明确搜索、滚动/分页、首中末或边界、有结果/无结果、清空恢复等覆盖策略；只有页面事实明确存在远程、异步、懒加载、分页加载或按需服务端来源时才可标记为动态，本地可搜索的有限下拉仍按有限集合全量覆盖；不得用“动态选项”替代实探。
- 每次选择都必须记录选择前状态、选择后页面变化、联动/依赖变化、结果分支、恢复/清空结果和独立证据；无页面变化、无联动也要明确记录。每行同时填写 `证据路径` 和可复核的 `证据定位`（截图区域、DOM/trace 步骤或视频时间点），同一控件的每个选项必须使用唯一的 `(证据路径, 证据定位)` 组合；同一视频或 trace 可以复用路径，但定位不得相同。
- `page-discovery.csv` 不能只写“查看选项”“展开下拉内容”“全部选项”；必须写清实际选择的选项值、选择后触发的页面变化、列表刷新、按钮禁用/启用、后续下拉变化或明确“无联动变化”。
- 选项汇总写入 `page-discovery.csv` 的 `选项取值/输入值` 和 `联动/依赖变化`；逐项结果写入 `selection-option-observations.csv`，有限集合每个选项一行，且选项序号必须连续、记录数必须等于可用选项总数。每行 `预期结果锚点` 必须取自真实观察到的稳定页面结果，并原样进入该选项精确关联用例的预期结果，禁止只落选项值后改写成通用结论。

## 风险问题页面可验证性

- `risk-confirmation.csv` 中真实风险必须填写 `页面可验证性`、`页面验证动作`、`页面验证结果` 和 `不可验证/外部依赖原因`。
- `页面可验证性=可直接验证`、未实际执行页面验证、缺少结果或证据时，流水线必须退回 discovery，不得进入用户确认。
- 已完成页面实探后，只有页面和现有产品事实仍无法解释的业务语义、验收口径或外部依赖含义，才进入风险确认。账号/验证码、必要权限、隔离环境、外部服务或无法安全创建的必要测试数据属于 `EXTERNAL_BLOCKED`：保留在 discovery，等待条件恢复后继续实探，不能记为已覆盖分支或风险确认结论。
- “只展开/仅查看/看到选项但未逐项点击”“权限未知/待确认”不是已覆盖，也不能伪装成风险项；真实禁用、无权限、置灰、403、不展示或外部服务不可用必须先实际尝试并记录具体结果与独立证据。数据不足时优先安全创建本次测试数据，仍无法补齐才进入 `EXTERNAL_BLOCKED`。
- 一个风险关联多个元素时使用逗号或分号分隔，并确保每个元素都能在同一页面的 `page-discovery.csv` 中精确匹配。

## 分页类控件

- 分页是复合控件，禁止只用一行“分页控件展示正常”或“共 N 条/每页 N 条”视为覆盖完成。
- 如果页面存在分页，`page-discovery.csv` 至少拆成每页条数下拉、上一页/下一页或页码跳转、首页/末页/边界页或禁用态三类元素。
- 每页条数属于有限集合时必须覆盖每个选项：可选项实际选择，真实禁用项尝试并记录阻塞证据；逐项观察列表刷新、总数、当前页、空页、末页数据不足一页和筛选条件变化后的分页重置。
- 功能测试用例必须覆盖条数切换、翻页、边界页、无数据或末页、筛选/搜索后分页重置和数据一致性。
- 当前数据不足以启用下一页、末页、页码跳转或每页条数真实效果时，不得生成声称这些功能已验证的功能用例；先使用安全的本次创建测试数据补齐条件。无法安全造数时保持未覆盖并阻塞 cases/delivery，而不是把“仅做设计”当作已实探。

## 输入类控件

- 输入框、搜索框、文本域、数字框、日期框、URL/地址、端口、邮箱、手机号、名称、编码等输入类控件不得只观察字段存在。
- 必须实际输入正常、异常、边界或用户提供的测试数据，触发查询、下一步、测试连接、预览、保存前校验等后续动作。
- 必须记录真实提示、字段校验位置、后续页面、结果分支/后续状态、禁用态和可恢复路径。
- 结果写入 `page-discovery.csv` 的 `选项取值/输入值`、`预期/观察行为`、`结果分支/后续状态`。

## 新增类流程

- 新增、创建、添加、新建、保存、提交、下一步、完成、测试连接等新增类流程必须实填实走。
- 在安全边界内实际填写测试数据并触发后续动作。
- 本次会落库或改变状态的数据必须使用 `AI_TEST`、`CODEX_TEST`、当前日期、任务编号或用户明确提供的测试数据标识；必须记录测试数据名称、编码或可检索关键字段。
- 真实新增成功后必须在 `test-data-lifecycle.csv` 记录测试数据ID/名称、创建入口、创建结果、查看结果、编辑前后值、配置生效验证点、删除取消结果、删除确认结果和清理状态。
- 成功时必须进入详情页、下一级页面或后续配置页继续观察可点击/可输入/可测试功能。
- 失败时必须记录真实失败提示、停留页面、字段校验位置、可恢复路径和是否允许重新提交。
- 新增成功后必须继续覆盖本次新增数据的查看、编辑、删除取消、删除确认或清理路径；如果删除会影响后续测试，必须记录保留原因和清理建议。

## 既有数据只读深探

- 对既有数据必须主动做只读深探：打开详情、进入编辑页观察字段和联动、尝试点击删除/停用/提交等危险按钮以打开确认弹窗或保存前校验。
- 不得点击最终确认、保存、提交或改变既有数据状态。
- 必须记录确认弹窗、二次确认、权限提示、取消路径、返回路径。
- 可以复制既有数据的非敏感字段，改名或改编码为带 `AI_TEST`、`CODEX_TEST`、当前日期或任务编号的新数据后新增，再用新数据探索后续页面。

## 变更类证据

- 编辑、删除、启用、停用、发布、下线、分配、转移等敏感或变更类操作，只有作用于本次创建且带测试标识的数据时才允许最终保存或确认。
- 对本次新增数据执行编辑时，必须记录编辑前值、编辑后值、保存结果、列表/详情回显和恢复或清理路径。
- `test-data-lifecycle.csv` 必须按 `交互实例ID` 精确匹配 mutation plan；编辑、配置、状态变更或删除使用本次创建数据时，必须与创建行复用同一 `测试数据ID/名称` 和完全相同的 `创建步骤关联用例`，禁止用无关创建对象为另一对象的修改提供假闭环。
- 对本次新增数据执行删除时，必须分别记录删除取消后数据不变、删除确认后列表不再展示或搜索不到。
- 对既有数据只能进入编辑页观察、触发保存前校验、打开删除确认弹窗并取消/关闭，步骤和预期必须明确“不保存、不提交、不最终确认、数据不变”。

## 配置项生效检查

- 认证类型、认证密钥、服务地址、工作流选择、标签、开关、状态、权限、默认值、变量、模型/智能体/工具配置、路由/接入方式等配置项，不能只验证保存按钮和成功提示。
- 配置项保存类路径必须检查保存后回显和实际生效：保存后返回列表或详情，重新打开查看/编辑页检查回显，再触发依赖功能、测试连接、预览、调用或关联流程验证配置是否生效。
- 如果配置生效依赖其他模块、接口、监控或造数，必须在 `element-case-plan.csv` 标记 `是否涉及配置生效=是`，并在风险与待确认问题或性能/自动化建议中说明补充验证条件。
- Discovery Agent 返回 `EXTERNAL_BLOCKED` 只表示真实外部条件尚未满足；恢复后仍必须继续完成原交互。页面可验证缺口、逐选项缺口或 CRUD 生效缺口必须形成指向 discovery 的结构化返工，不得进入 Risk Arbiter。
