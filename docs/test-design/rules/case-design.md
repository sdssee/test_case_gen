# 测试计划与用例规则

## 独立功能与计划

从 facts 动态识别具有独立触发和可观察结果的用户功能，不依赖预设功能名称或固定控件清单。页面上实际出现的查询、状态切换、内容浏览、详情、数据维护等能力均按同一规则判断；单个控件、具体选项和普通字段本身不自动等同于独立功能。

运行时从facts生成只读计划骨架：除空值、边界、有限选项等元素级提示外，直接带入实测动作、关联分支、稳定结果锚点、Case级测试点建议、主验证目标建议和已观察状态。模型据此为每个功能一次填写 `design_context`（用户目标、角色、业务价值、验收标准、业务规则、依赖、后置影响、真实依据）和 `automation_profile`，并在唯一 `check_assignments` 中分配检查点。每个Case确认唯一 `verification_focus`，明确一个单独分支或一个实测有效关联及其可观察效果；只有 `branch_bindings` 中的控件可以共同完成关联覆盖，普通辅助使用不能替代独立覆盖。系统从同一账本派生事实引用、测试点和元素覆盖，不维护独立DFX或关联矩阵账本。

页面交互用例的自动化层级归一为UI；接口、命令行或组件层只能用于对应入口的测试，不替代页面Case。自动化画像中同类稳定性风险由运行时聚合提升为一个风险并保留受影响功能。按钮、提交动作或触发引用本身不是性能事实；只有实测加载、异步、长任务、超时、耗时或需求性能目标时，运行时生成一条功能范围内共享的轻量响应场景，不按Case复制；只有确无可观察性能行为或具体风险时才保留不适用理由。

一个页面事务可以连续实探多个选项或输入类别。`required_case_branches` 按实际检查形成受约束场景：同一检查明确绑定并共同产生结果的分支进入一个关联Case，不同关联签名分别成Case；没有关联时仍逐分支成Case。有限选项和有效输入类使用 baseline，空值、无效格式、边界等使用其预声明 DFX 策略。普通辅助使用不能替代独立覆盖，也不默认展开跨维度组合。

## 用例正文

- 只按 `case-plan.json` 的稳定 Case ID、`verification_focus` 和功能顺序写作，不临时发明功能、场景和预期。
- 计划标题只写具体场景，不重复功能名称；最终标题由系统按“功能名称-具体场景”形成且不得为空。Case级 `test_point` 由实测关联自动派生，不要求模型重复填写。
- 模型只写业务步骤；运行时根据页面事实统一注入第一步完整实际菜单路径，使用 `-` 连接且不逐级拆分菜单。
- 前置条件只描述角色、环境、状态和数据；菜单导航写入步骤。
- 使用具体且脱敏的数据，不写“输入测试数据”、自然语言地址或 `<有效值>` 等占位描述；敏感或环境相关值使用 `TEST_*` 命名引用，并在前置条件说明同名受控来源。
- 模型为每个业务步骤写 `{action, expected}`；动作必须包含同一事实检查中的完整输入、具体选项和触发过程。系统按计划分配顺序自动注入 `source_check`、`test_point` 和主验证目标。一个步骤对应一个检查点，预期必须表达本Case的关联行为并优先满足同一结果锚点中的 `stable_tokens`，不得借用其他关联结果，也不得把偶发计数、进度等样本值固化为稳定结论。导航步骤不绑定检查点，内部来源不导出Excel。
- 每条用例填写 `automation_value`、`automation_priority`；只有与功能默认画像不同才填写 `automation_override`，不复制整套自动化说明。
- 同功能用例连续排列。公共导航允许重复，核心操作和预期分别都应体现主验证分支，不能只改标题或数据便复用相同动作/相同预期。
- 计划和用例均在各自单文件中按功能 upsert；再次提交同一功能时替换该功能块，其他功能保持不变，精确重复提交不产生重复内容。
- 禁止截图要求、UID、DOM、选择器、事实编号和页面工具术语。

CRUD、编辑和配置用例必须写出提交后的查询、重开回显、实际效果和恢复/清理。无效分支写清具体拦截与数据未变化，不能编造成成功结果。
