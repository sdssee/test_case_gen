# Codex Project Instructions

本仓库是测试设计与测试用例生成规范包，不是传统应用代码项目。Codex 处理本仓库或复制到业务项目后，应把本文件作为项目级执行说明。

## 核心目标

- 基于需求文档、用户故事、接口文档、页面截图、原型、可访问页面、PR Diff、缺陷单或已有用例，生成结构化测试设计。
- 正式交付物优先使用 `docs/test-design/codebuddy-test-design-template.xlsx`。
- 输出必须覆盖功能测试、性能测试、风险与待确认问题、自动化建议。
- 涉及页面、截图、原型或可访问系统时，必须输出并维护 `页面元素覆盖清单`。
- `页面元素覆盖清单` 只是覆盖追踪矩阵，不是测试用例 Sheet；所有功能测试用例必须写入 `功能测试用例` Sheet，所有性能测试场景必须写入 `性能测试设计` Sheet。
- 不再向正式测试设计 Excel 中新增 `测试系统导入用例` Sheet；需要导入测试系统时，必须复制 `docs/test-design/测试用例模板.xlsx` 生成独立导入文件，并把从 `功能测试用例` 派生的内容填充到该副本中。
- 测试资产事实必须沉淀到项目文件，不依赖 AI 对话记忆。客户交付件放在 `docs/test-design/current/` 或 `docs/test-design/deliverables/`；内部产品级测试资产库放在 `docs/test-assets/`，其中 `docs/test-assets/product-map.xlsx` 是产品测试知识图谱主入口，不作为默认客户交付件。

## 使用现有规范

Codex 应优先读取并遵守以下文件：

- `CODEBUDDY.md`
- `.codebuddy/skills/test-design/SKILL.md`
- `.codebuddy/.rules/test-design-rule.mdc`
- `.codebuddy/rules/test-design-rule.md`
- `docs/test-design/excel-template-spec.md`

这些文件虽然位于 CodeBuddy 目录中，但内容同样作为 Codex 的测试设计规范。

## 文档与页面融合规则

- 每次生成或补充测试用例前，必须先读取 `docs/test-assets/product-map.xlsx` 和用户指定依赖模块的归档测试设计；如果用户说明当前模块依赖某模块能力，应优先按产品版图中的归档文件路径读取对应测试设计。
- 生成正式测试用例前，必须向用户展示产品理解摘要，包括当前模块、依赖模块、关联业务对象、关联业务链路、可复用历史用例、预计新增范围和待确认问题；用户确认或补充后再生成正式交付件。
- 如果产品版图缺少用户提到的依赖模块，但 `docs/test-assets/modules/` 中存在相关归档文件，应读取归档文件并把缺失的能力、依赖或归档路径登记为待同步项。
- 不得仅凭模型记忆判断已有模块能力、已有用例、可复用测试数据或跨模块依赖。
- 当用户要求为某个模块生成测试设计时，正式写用例前必须先做模块级粗遍历，快速识别菜单入口、页面清单、核心功能点、主要业务对象、状态流转和跨模块依赖，形成产品理解摘要或模块理解摘要。
- 模块级粗遍历、菜单轮廓、页面清单和功能地图不是临时分析结果，必须沉淀到 `product-map.xlsx`：菜单和页面写入产品模块地图，页面入口和交互元素写入页面元素地图，业务对象和状态流转写入业务对象地图或业务链路地图，跨模块关系写入跨模块依赖关系，模块能力写入模块能力索引，并在变更记录中说明本次同步。
- 具体写测试用例时，必须在对应页面或功能点内做深遍历，覆盖所有可点击、可输入、可测试元素，包括按钮、链接、菜单、Tab、图标按钮、筛选、排序、分页、表格行操作、批量操作、输入框、下拉项、上传下载、弹窗、抽屉、开关、禁用态、空状态和错误态。
- 深遍历下拉框、级联选择、单选框、复选框、树选择、枚举筛选等选择类控件时，不得只展开查看选项；必须分别选择代表性选项，观察后续字段、二级/三级下拉、列表数据、按钮禁用态、校验提示、默认值、清空重置、搜索结果和提交前提示的变化。对存在前后依赖的控件，必须记录每个关键选项触发的联动/依赖变化；无联动也要明确记录为无联动变化。
- 深遍历输入框、搜索框、文本域、数字框、日期框、URL/地址、端口、邮箱、手机号、名称、编码等输入类控件时，不得只观察字段存在或只写“输入后校验”；必须实际输入正常、异常、边界或用户提供的测试数据，触发查询、下一步、测试连接、预览、保存前校验等后续动作，并记录真实提示、字段校验位置、后续页面、结果分支/后续状态、禁用态和可恢复路径。
- 遇到新增、创建、添加、新建、保存、提交、下一步、完成、测试连接等新增类流程时，必须实填实走，在安全边界内实际填写测试数据并触发后续动作；成功时必须进入详情页、下一级页面或后续配置页继续观察可点击/可输入/可测试功能，失败时必须记录真实失败提示、停留页面、字段校验位置、可恢复路径和是否允许重新提交。不得只停留在新增表单页推测后续测试范围。
- 大范围任务进入每个批次正式写测试用例前，如果存在可访问页面、原型或桌面窗口，必须使用浏览器能力或 computer use 遍历当前批次对应模块、页面域或业务链路的所有可点击/可交互功能点，并把发现结果写入页面元素覆盖清单和功能测试用例。
- 每一批测试设计都必须严格执行完整 test-design Skill 和 Rule，不得因为分批而降级；每批都必须覆盖功能测试、性能测试、异常流程、边界值、权限/角色、状态流转、数据一致性、兼容性/稳定性、风险与待确认问题、自动化建议和页面元素覆盖清单。
- 当任务范围超过一个最小标题时，禁止直接生成完整测试用例；必须先建立批次队列，并在 `docs/test-assets/batch-runs/<YYYYMMDD>_<任务标识>/` 创建或更新批次运行状态账本，包含 `batch-plan.md`、`batch-status.csv`、`batch-review.md`、`page-discovery.csv` 和 `artifacts/`。每批必须按最深可识别标题形成唯一 `最小标题路径`，完成页面元素覆盖、功能测试、性能测试、异常、边界、权限、状态、数据一致性、风险与待确认问题、自动化建议、资产回存和 `batch-status.csv` 覆盖质量自检后，才能进入下一批；`batch-status.csv` 必须记录最小标题路径、页面数、元素总数、已覆盖元素数、待确认元素数、功能用例数、性能场景数、异常用例数、边界用例数、权限/状态用例数和数据一致性用例数、导入文件路径和导入文件已生成。所有批次完成后只做最终汇总、跨模块汇总、回归范围、风险清单和客户总览，不得重新生成各批完整用例。
- 大范围任务禁止创建承载全量测试用例正文的单一中间文件，例如单个 Python、JSON、CSV、Markdown 或临时脚本文件；脚本只能用于当前批次的模板填充、格式转换或校验，并保存到本任务 artifacts/scripts/，不得把多个最小标题路径、多个批次或全产品测试用例先集中写入一个文件后再统一生成 Excel。
- 测试用例必须尽可能详细。每个测试点、每个页面元素（包括可点击/可输入/可选择/可上传下载/可展开收起/可批量操作/可行内操作的元素），都必须严格按 Skill 从不同测试方向展开：主流程、异常流程、边界值、权限/角色、状态流转、数据一致性、组合条件、禁用态/空状态/错误态、兼容性/稳定性、性能影响、审计/日志/通知、副作用和可恢复路径；不能只写一个笼统用例替代多个可验证方向。
- 业务逻辑从设计文档、需求说明、接口文档、用户故事、验收标准中抽取。
- 页面截图、原型或实际页面用于补齐具体入口、控件名称、点击路径、表单字段、弹窗、表格行操作、筛选、排序、分页、禁用态、空状态和错误态。
- 如果文档只描述大概功能，应结合页面入口补全测试场景。
- 如果页面存在文档未说明的入口，不得直接假设完整业务规则；应生成基础交互覆盖，并把业务规则、权限、数据影响或验收标准登记为待确认问题。
- 页面所有可点击/可交互元素必须关联测试用例、标记不适用、标记不测范围或登记待确认问题。

## Excel 交付要求

正式测试设计 Excel 默认包含：

1. `测试设计总览`
2. `需求用户故事拆解`
3. `测试场景矩阵`
4. `功能测试用例`
5. `性能测试设计`
6. `风险与待确认问题`
7. `自动化建议`
8. `页面元素覆盖清单`

功能测试用例中的 `前置条件`、`操作步骤`、`预期结果` 必须使用 `1. 2. 3.` 编号换行。`操作步骤` 不得默认执行人已经位于当前模块页面，必须从系统或项目入口开始写完整导航路径，例如登录系统、进入一级菜单、进入二级/三级菜单或打开目标页面，再描述具体控件操作和提交/校验动作。

功能测试用例中的 `用例标题` 以及测试系统导入文件中的 `测试用例名称` 必须正式、简洁、可检索，避免口语化、聊天式或操作随笔式表达；标题应能说明测试项，并优先体现业务对象、条件和预期行为。由于测试系统导入后没有独立 `功能点` 字段，`用例标题` 必须使用 `功能点-当前用例标题` 格式，例如 `搜索筛选-按智能体名称查询列表`；短横线前后不加空格，功能点为空时应先补齐功能点，不得省略前缀。

`页面元素覆盖清单` 只记录页面元素、业务依据、覆盖状态、发现方式、素材来源和关联的 `覆盖用例 ID`。不得在该 Sheet 中编写独立测试用例、操作步骤、测试数据或完整预期结果正文。

测试系统导入内容必须写入 `docs/test-design/测试用例模板.xlsx` 的副本，字段必须严格匹配该模板：一级到五级模块系统编号/名称、其他模块系统编号/名称、测试用例系统编号、测试用例序号、测试用例名称、测试步骤描述、测试步骤预期结果、测试类型、测试用例级别、执行方式、测试用例说明、前置条件、维护人、标签、备注、作者。

不得修改 `docs/test-design/测试用例模板.xlsx` 原模板文件；每次生成导入文件时都应复制模板后填充副本，从而保留下拉框、必填字段样式、标红字段和其他测试系统导入约束。

测试系统模板中的红色字段和必填字段必须按每条测试用例动态生成，不得固定写示例值或通用占位值。红色字段包括 `测试用例名称`、`测试步骤描述`、`测试步骤预期结果`、`测试类型`、`测试用例级别`、`执行方式`；必填字段至少包括 `测试用例名称`、`测试类型`、`测试用例级别`、`执行方式`。

测试系统模板中带下拉框的字段只能填写下拉框允许的内容。当前下拉字段为：`测试类型`（功能测试、性能规格测试、可靠性测试、兼容性测试、可维护性测试、安全性测试、易用性测试）、`测试用例级别`（L1、L2、L3、L4）、`执行方式`（自动化、手动）。自动生成字段不需要填写，不得编造值，例如 `测试用例系统编号`、`作者`、`创建时间` 等。

`执行方式` 表示测试系统中该条用例当前如何执行，不表示“未来是否适合自动化”。默认填写 `手动`。只有同时满足以下条件时才允许填写 `自动化`：存在已实现、可运行、可维护的自动化脚本/API/流水线任务；该自动化资产能覆盖本条用例的主要校验点；本次交付明确要求将该用例按自动化执行导入或关联自动化资产。仅页面实探、AI 点击验证、CodeBuddy/Codex 辅助生成、或在 `自动化建议` 中判断“适合自动化”，都不能把 `执行方式` 写成 `自动化`。

## 工具使用

- 有可访问页面时，优先使用浏览器能力观察页面；只能通过桌面窗口或既有登录态访问时，可使用 computer use。
- 如果用户提供测试数据，优先用这些数据进行页面实探和用例设计；正常数据用于校准成功路径预期，异常/边界数据用于补充负向测试。
- 使用用户提供测试数据时，若数据明确用于本次测试并带有测试标识，可以执行新增、保存、编辑、删除、状态变更等敏感或高风险操作；若数据会修改已有数据或无法确认归属，只能查看、填入观察校验或设计用例，不得保存提交。敏感数据必须在最终用例中脱敏或替换为占位符。
- 页面实探得到的实际提示文案、字段限制、联动效果、跳转结果、状态变化和错误信息，应写入 `操作步骤` 和 `预期结果`。
- 页面实探得到的下拉选项、实际选择值、级联/依赖变化、选项切换后的字段或列表变化，应写入 `page-discovery.csv` 的 `选项取值/输入值`、`联动/依赖变化`，并反映到功能测试用例的操作步骤和预期结果中。
- 页面实探得到的输入类控件实际输入值、校验反馈、后续页面和结果分支，应写入 `page-discovery.csv` 的 `选项取值/输入值`、`预期/观察行为`、`结果分支/后续状态`，并反映到功能测试用例的操作步骤和预期结果中。
- 页面实探得到的新增类流程提交数据、成功/失败结果、进入的下一级页面或失败后的停留状态，应写入 `page-discovery.csv` 的 `选项取值/输入值`、`预期/观察行为`、`结果分支/后续状态`，并继续补充下一级页面的页面元素覆盖清单和功能测试用例。
- 页面实探时，只能新增、修改、删除或变更自己创建且带测试标识的数据；对页面已有数据可以查看、搜索、筛选、排序、分页、打开详情，或进入编辑/修改页面观察字段、校验、联动、多级子菜单和保存前提示，但不得保存、提交或改变已有数据。
- 新增测试数据必须带明显标识，例如 `AI_TEST`、`CODEX_TEST`、当前日期或任务编号；无法确认是否自己创建的数据，一律按已有数据处理。
- 对本次创建且带测试标识的数据，可以执行删除、审批、启停、发布、批量处理、外部调用等敏感操作以确认真实页面反馈；测试用例必须明确“仅操作本次创建的测试数据”。对已有数据仍只允许查看或进入编辑页观察，不保存不提交。
- 修改 `.xlsx` 文件时，应保持现有样式和字段结构，修改后检查 Sheet、表头、换行、筛选和公式错误。
- 生成测试系统导入文件时，除字段值合法外，还必须保留模板中的测试系统下拉字段、必填字段样式和自动生成字段空值；不得通过新建空白 Sheet 或手工仿制表头替代模板副本。
- 生成正式测试设计 Excel 后，应使用 `scripts/validate-test-design-deliverable.ps1 -WorkbookPath <测试设计.xlsx>` 校验交付件结构、页面元素覆盖关系、标题格式、编号步骤和性能设计；大范围任务还应传入 `-BatchStatusPath <batch-status.csv>` 校验批次状态与交付件一致，并强制读取同级 `page-discovery.csv` 与 `docs/test-assets/product-map.xlsx` 做产品版图同步校验；也可以显式传入 `-ProductMapPath docs/test-assets/product-map.xlsx -PageDiscoveryPath <page-discovery.csv>`，校验页面实探、正式 Excel 和产品版图之间的最小标题路径、页面元素、关联用例、用例资产索引和变更记录是否同步。

## 用例编排

- 功能测试用例必须按模块、页面、业务流程和小功能块组织。
- 同一小功能块的用例必须连续放在附近，例如新增、查看、编辑、删除、搜索、筛选、分页、导入、导出、审批等分别成组。
- 每个小功能块内部再按主流程、异常流程、边界值、权限/状态、数据一致性、兼容性或稳定性排序。
- 跨模块用例只保留一个主归属模块；依赖模块通过 `product-map.xlsx` 中的业务对象地图、业务链路地图、用例资产索引、模块能力索引、跨模块依赖关系、标签、备注和关联用例 ID 追踪。当前模块依赖已有模块能力时，优先引用已有用例 ID 作为前置条件，不重复复制已有用例。
- 每次生成、评审或人工修订后的最终版测试设计，应回存到 `docs/test-assets/modules/`；测试系统导入文件副本回存到 `docs/test-assets/imports/`；客户交付件放在 `docs/test-design/current/` 或 `docs/test-design/deliverables/`；并同步更新 `product-map.xlsx` 的产品模块地图、业务对象地图、业务链路地图、页面元素地图、用例资产索引、模块能力索引、跨模块依赖关系、可复用测试数据、变更影响分析和变更记录。

## Git 约定

- 如果修改已完成且验证通过，默认提交到当前分支并推送到 `origin`。

## 外网到内网升级约定

- 普通框架升级以脚本升级为主、手动确认兜底：外网使用 `scripts/new-framework-upgrade-package.ps1` 生成升级包，内网使用 `scripts/upgrade-framework.ps1 -PackagePath <升级包>` 应用升级包。
- `docs/test-assets/`、`docs/test-design/current/`、`docs/test-design/deliverables/` 是受保护目录，普通框架升级不得覆盖、删除或清空。标识：PROTECTED_ASSET_DIRS。
- `VERSION` 中的 `framework_version` 表示框架版本，`asset_schema_version` 表示内部资产结构版本；只有资产结构版本或 `product-map.xlsx` 结构变化时才需要迁移。
- 资产结构升级必须通过升级清单、校验脚本和迁移脚本发现并处理；迁移脚本只能读取旧资产并增量补齐，不能用空模板覆盖内网真实资产。
- `docs/test-assets/modules/` 和 `docs/test-assets/imports/` 中的历史 Excel 默认作为历史快照保留，不因普通框架升级而批量重写。


- 分批必须按当前产品或模块可识别的最深标题级别执行，例如一级标题、二级标题、三级标题、四级标题等，哪个标题级别最小就以哪个最小标题作为一个批次。每个已通过批次只能覆盖 1 个最小标题路径，`最小标题路径` 使用 `一级>二级>三级>四级` 形式记录唯一叶子节点；禁止合并多个最小标题，禁止再拆分一个最小标题为多个批次。已通过批次的导入文件路径必须真实存在，并能与归档测试设计逐个匹配校验。
- 当任务范围是全产品、多个一级模块或大模块时，必须先遍历一级菜单、二级菜单、三级菜单及更深层标题，拿到菜单轮廓、页面清单和功能地图后输出分批设计计划，不得一次性生成完整测试用例；各批按最小标题路径执行，所有批次完成后再做跨模块汇总。
- 当任务范围超过一个最小标题时，必须建立批次队列并按最小标题路径逐批执行。
