# 测试设计硬规则

1. 单会话、单浏览器上下文顺序执行；不得创建或调用多 Agent，不得依赖页面操作 Hook。
2. 用户直接调用 Skill；运行目录自动绑定或恢复，不出现独立初始化阶段。
3. 默认全量深探并开放发现未知控件；扫描和事务交替，不建立逐元素义务队列。
4. 元素登记时先由 DFX 策略根据必填、格式、边界、有限选项和配置基线生成 `exploration_requirements`，再执行交互；输入至少实测正常值，必填输入同时实测空值，明确格式/边界时实测相应无效和边界分支；有限选项逐项实际选择；需要按钮触发时检查必须包含完整触发动作；CRUD、普通编辑和单因素配置必须验证提交、持久化回显、实际效果和恢复/清理。
5. 页面可验证的问题必须自行操作；只有外部业务语义、权限、数据或环境阻塞进入 `open_items`。
6. 证据是可选诊断，不是事实、计划、Review 或交付的输入，不默认创建证据目录。
7. 独立功能先有基线用例；DFX 在计划阶段按事实适用性展开，不使用固定矩阵或固定用例数。
8. 一个用例归属一个功能；辅助使用其他控件不能替代该控件所属功能的独立覆盖。
9. 已完成的实探检查立即写入；每个检查记录 `used_element_refs` 和适用的 `input_class`。未执行的 DFX 分支保留在既定清单中并持续提示，不拒绝已完成事实，也不在 checkpoint 临时新增义务；声明但未使用的控件、缺失触发动作或CRUD/配置闭环仍属于事务结构错误。facts按页面checkpoint刷新且可自动重建。计划只维护唯一 `check_assignments`，引用、元素覆盖和DFX关联由系统派生。
10. 每条用例第一步使用页面事实中的完整实际菜单路径；其余步骤由模型写action+expected，系统按检查点分配自动注入source_check。预期通过结构化结果锚点校验，内部来源不导出。
11. 禁止把截图、UID、DOM、选择器、内部编号、工具操作和占位数据写入可执行用例。
12. 同功能用例连续排列；公共导航可以重复，核心操作、数据和预期不得整段重复。
13. Review 只做一次跨产物审计，不返回笼统 passed/failed，不触发全量返工或自动重试。
14. 正式测试设计与测试系统导入 Excel 从各自模板、同一结构化用例源独立生成；不得出现空标题、中间空行、跨Sheet字段不一致或因行高不足造成的文本截断。
15. 既有数据只读；只修改或删除本轮带 `AI_TEST`、`CODEX_TEST` 或用户明确标识的测试对象。
16. 保护 `docs/test-assets/`、`docs/test-design/current/`、`docs/test-design/deliverables/` 中已有用户资产。
17. 内部执行只使用标准CLI和JSON负载，禁止临时Python编排脚本；运行目录不得逃逸当前项目根目录。
