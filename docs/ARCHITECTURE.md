# 测试设计架构

## 总体流程

```text
直接调用Skill
→ 页面能力扫描
→ 连续页面深探
→ 用户确认与增量补探
→ DFX驱动Case规划
→ 多JSON分片
→ 双Excel原子交付
→ 一次最终Review
```

## 页面与事实

- 页面是已实现功能的主要事实来源；需求、原型、文档和模型推断用于参考。
- DOM、可访问性树、可见文案、悬停状态和实际操作共同识别动态页面能力。
- 页面能验证的问题由模型操作。用户确认后，对相关页面分支增量补探，不重复已实测内容。
- `page-discovery.csv`使用`已实测/页面观察/DFX设计/待确认`区分来源，推断不得冒充实测。

## Case与DFX

- 每个有限选项、每个有效输入等价类分别形成baseline Case，不默认做组合积。
- 元素在其他Case中被辅助使用不替代自身主验证Case。
- CRUD及可选配置使用本轮数据按单因素完成真实保存、回显、生效、取消和删除验证。
- DFX在规划时驱动适用扩展；性能场景进入性能Sheet，阈值注明需求、基线、建议或待确认来源。

## 分片与质量

- 按功能块保留多个小JSON，避免单次写入过大。
- 每个分片写入前完成编号、导航、排序、具体步骤/预期和确定性重复检查。
- 普通选择、输入和查询以具体结果闭环；状态变更事务按业务结果闭环。
- 禁止临时`fix_*.py`、逐元素义务队列、Hook和自动返工循环。

## 批次与恢复

- 页面运行保留`batch-status.csv`、`page-discovery.csv`和JSON分片；诊断证据可选且不参与后续判断。
- 单页面不强制批次计划；多页面、多模块或大范围任务增加精简`batch-plan.md`。
- 不预创建空白Review。全部产物完成后只创建一次`final-review.md`。

## Excel交付

- 正式工作簿固定8个Sheet，导入工作簿独立交付。
- 两个Excel来自同一次分片汇总。
- 文件名使用真实1-N级菜单路径；导入字段补位不能改变文件名。
- 模板填充前清空示例数据；两个临时文件全部通过后可回滚发布。
- 最终路径直接使用工具返回值。

## 数据与操作边界

- 内网IP、URL、账号、密码、Token、密钥、Cookie、部署路径和测试载荷允许原样流通，不做敏感内容校验或脱敏。
- 只能改变本轮创建的测试数据；共享环境变更后恢复。
- 框架升级保护`docs/test-assets/`、`docs/test-design/current/`和`docs/test-design/deliverables/`。
