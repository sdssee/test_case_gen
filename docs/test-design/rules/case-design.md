# 测试计划与用例规则

## 独立功能与计划

从 facts 动态识别具有独立触发和可观察结果的用户功能，不依赖预设功能名称或固定控件清单。页面上实际出现的查询、状态切换、内容浏览、详情、数据维护等能力均按同一规则判断；单个控件、具体选项和普通字段本身不自动等同于独立功能。

运行时从facts生成只读计划骨架：空值、边界、有限选项等是元素级提示，并只关联具有相应 `input_class`、`option_value` 或DFX标签的实测检查；持久化和恢复是事务级提示；权限和业务状态是功能级提示。模型据此补充Case意图，并在唯一 `check_assignments` 中把每个检查点分配给Case，或明确进入performance、risk、not_applicable并填写原因。系统从该账本派生主验证和辅助使用的元素覆盖关系，不再维护独立DFX账本。

## 用例正文

- 只按 `case-plan.json` 的稳定 Case ID 和功能顺序写作，不临时发明功能、场景和预期。
- 标题使用“功能点-具体场景”，不得为空。
- 每条用例第一步使用完整实际菜单路径，例如“进入告警管理-告警列表”，不逐级拆分菜单。
- 前置条件只描述角色、环境、状态和数据；菜单导航写入步骤。
- 使用具体且脱敏的数据，不写“输入测试数据”等占位描述。
- 模型为每个业务步骤写 `{action, expected}`；动作必须包含事实检查中的完整输入、选择和触发过程，不能省略产生结果所需的按钮。系统按计划分配顺序自动注入 `source_check`。一个步骤对应一个检查点，预期满足结构化结果锚点中的明确tokens或value即可，不要求复制目标、字段或事实原句。导航步骤不绑定检查点，内部来源不导出Excel。
- 同功能用例连续排列。公共导航允许重复，核心操作、数据和预期不得整段重复。
- 禁止截图要求、UID、DOM、选择器、事实编号和页面工具术语。

CRUD、编辑和配置用例必须写出提交后的查询、重开回显、实际效果和恢复/清理。无效分支写清具体拦截与数据未变化，不能编造成成功结果。
