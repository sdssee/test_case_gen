# 测试计划与用例规则

## 独立功能与计划

从 facts 动态识别具有独立触发和可观察结果的用户功能，不依赖预设功能名称或固定控件清单。页面上实际出现的查询、状态切换、内容浏览、详情、数据维护等能力均按同一规则判断；单个控件、具体选项和普通字段本身不自动等同于独立功能。

运行时先从 facts 生成只读计划骨架，列出功能、所属页面、元素、事务检查点和按事实推导的 DFX 提示。模型在骨架上补充基线意图、适用 DFX 和检查点分配；每条提示在 `dfx_decisions` 中绑定 Case，或以 performance、risk、not_applicable 说明原因，避免把 DFX 留到 Review 补做。一个事务的多个检查点可以归入一条用例，也可以按不同测试意图拆分；不得按点击数固定生成用例。每个事务检查点必须分配给某条 Case，或明确进入 performance、risk、not_applicable 并填写原因。

## 用例正文

- 只按 `case-plan.json` 的稳定 Case ID 和功能顺序写作，不临时发明功能、场景和预期。
- 标题使用“功能点-具体场景”，不得为空。
- 每条用例第一步使用完整实际菜单路径，例如“进入告警管理-告警列表”，不逐级拆分菜单。
- 前置条件只描述角色、环境、状态和数据；菜单导航写入步骤。
- 使用具体且脱敏的数据，不写“输入测试数据”等占位描述。
- 每个业务步骤使用 `{action, expected, source_check}` 配对；一个步骤只对应一个事务检查点，预期保留该检查点的实际结果锚点。导航步骤不带 `source_check`，内部来源不导出 Excel。
- 同功能用例连续排列。公共导航允许重复，核心操作、数据和预期不得整段重复。
- 禁止截图要求、UID、DOM、选择器、事实编号和页面工具术语。

CRUD、编辑和配置用例必须写出提交后的查询、重开回显、实际效果和恢复/清理。无效分支写清具体拦截与数据未变化，不能编造成成功结果。
