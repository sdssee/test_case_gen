# CodeBuddy Rule：测试设计硬性质量规则

## 最高优先级规则

1. 正式测试设计交付物必须是 Excel。
2. 任何测试设计都必须包含性能测试内容。
3. 敏捷用户故事每条至少生成 10 条功能测试用例。
4. `前置条件`、`操作步骤`、`预期结果` 必须使用 `1. 2. 3.` 换行分条；`操作步骤` 不得默认执行人已经位于当前模块页面，必须从系统或项目入口开始写完整导航路径。
5. 用例必须可执行、可验证、可追踪。
6. 如果输入包含页面、截图、原型或可访问地址，必须覆盖页面所有可点击/可交互功能。
7. 无法覆盖的页面元素必须登记为不测范围、待确认问题或不适用，并说明原因。
8. 页面业务逻辑必须结合设计文档、需求说明、接口文档、验收标准等资料推导；页面实探主要用于补全具体入口、控件、交互路径和可见状态。
9. `页面元素覆盖清单` 只是覆盖追踪矩阵，不是测试用例 Sheet；所有功能测试用例必须写入 `功能测试用例` Sheet，所有性能测试场景必须写入 `性能测试设计` Sheet。
10. 使用浏览器或 computer use 实探页面时，只能新增、修改、删除或变更自己创建且带测试标识的数据；对页面已有数据可以查看、搜索、筛选、排序、分页、打开详情，或进入编辑/修改页面观察字段、校验、联动、多级子菜单和保存前提示，但不得保存、提交或改变已有数据。
11. 功能测试用例必须按模块、页面、业务流程和小功能块连续编排；同一小功能块的用例必须放在附近。
12. 如果用户提供测试数据，必须优先用于页面实探和用例设计；正常数据用于校准成功路径预期，异常/边界数据用于补充负向测试，最终用例中的敏感数据必须脱敏。
13. 正式测试设计 Excel 不再新增 `测试系统导入用例` Sheet；需要导入测试系统时，必须复制 `docs/test-design/测试用例模板.xlsx` 生成独立导入文件，并从 `功能测试用例` 派生内容填充该副本。
14. 测试系统导入文件中的 `执行方式` 默认填写 `手动`；只有已存在可运行、可维护并覆盖该用例主要校验点的自动化资产，且本次交付明确按自动化导入或关联自动化资产时，才允许填写 `自动化`。
15. 测试资产事实必须保存在项目文件中，不依赖 AI 对话记忆；生成前读取 `docs/test-assets/product-map.xlsx`，正式生成前展示产品理解摘要，生成后回存 `docs/test-assets/modules/`、`docs/test-assets/imports/` 并更新产品版图。客户交付件放在 `docs/test-design/current/` 或 `docs/test-design/deliverables/`，内部产品版图不作为默认客户交付件。

## 分析先行规则

生成测试用例前，必须先完成：

- 需求理解
- 测试范围
- 不测范围
- 风险点
- 假设条件
- 待确认问题
- 页面元素覆盖清单（仅页面/截图/原型/可访问系统场景）
- 业务依据与页面交互的对应关系（仅混合材料场景）

不得在未分析需求的情况下直接输出测试用例。

## 测试资产归档与产品版图规则

- 归档规范见 `docs/test-design/archive-and-index-guidelines.md`。
- 内部产品测试知识图谱主入口为 `docs/test-assets/product-map.xlsx`。
- 客户交付件和内部维护资产必须分离；`product-map.xlsx` 不作为默认客户交付件。
- 生成或补充测试用例前，必须读取产品版图。
- 用户指定依赖模块时，必须优先读取产品版图中登记的依赖模块归档测试设计。
- 当用户要求为某个模块生成测试设计时，正式写用例前必须先做模块级粗遍历，快速识别菜单入口、页面清单、核心功能点、主要业务对象、状态流转和跨模块依赖。
- 正式生成前必须向用户展示产品理解摘要或模块理解摘要，包括当前模块、依赖模块、业务对象、业务链路、可复用历史用例、预计新增范围和待确认问题。
- 模块级粗遍历、菜单轮廓、页面清单和功能地图不是临时分析结果，必须沉淀到 `product-map.xlsx` 的产品模块地图、页面元素地图、业务对象地图、业务链路地图、模块能力索引、跨模块依赖关系和变更记录。
- 大范围任务进入每个批次正式写测试用例前，如果存在可访问页面、原型或桌面窗口，必须使用浏览器能力或 computer use 遍历当前批次对应模块、页面域或业务链路的所有可点击/可交互功能点，并把发现结果写入页面元素覆盖清单和功能测试用例。
- 具体写测试用例时，必须在对应页面或功能点内做深遍历，覆盖所有可点击、可输入、可测试元素。
- 每一批测试设计都必须严格执行完整 test-design Skill 和 Rule，不得因为分批而降级；每批都必须覆盖功能测试、性能测试、异常流程、边界值、权限/角色、状态流转、数据一致性、兼容性/稳定性、风险与待确认问题、自动化建议和页面元素覆盖清单。
- 当任务范围超过一个三级菜单/页面域时，禁止直接生成完整测试用例；必须先建立批次队列，并在 `docs/test-assets/batch-runs/<YYYYMMDD>_<任务标识>/` 创建或更新批次运行状态账本，包含 `batch-plan.md`、`batch-status.csv`、`batch-review.md`、`page-discovery.csv` 和 `artifacts/`。每批完成页面元素覆盖、功能测试、性能测试、异常、边界、权限、状态、数据一致性、风险与待确认问题、自动化建议、资产回存和 `batch-status.csv` 覆盖质量自检后，才能进入下一批。`batch-status.csv` 必须记录页面数、元素总数、已覆盖元素数、待确认元素数、功能用例数、性能场景数、异常用例数、边界用例数、权限/状态用例数和数据一致性用例数、导入文件路径和导入文件已生成。所有批次完成后只做最终汇总、跨模块汇总、回归范围、风险清单和客户总览，不得重新生成各批完整用例。
- 大范围任务禁止创建承载全量测试用例正文的单一中间文件，例如单个 Python、JSON、CSV、Markdown 或临时脚本文件；脚本只能用于当前批次的模板填充、格式转换或校验，并保存到本任务 artifacts/scripts/，不得把多个三级菜单/页面域、多个批次或全产品测试用例先集中写入一个文件后再统一生成 Excel。
- 测试用例必须尽可能详细。每个测试点、每个页面元素（包括可点击/可输入/可选择/可上传下载/可展开收起/可批量操作/可行内操作的元素），都必须严格按 Skill 从不同测试方向展开：主流程、异常流程、边界值、权限/角色、状态流转、数据一致性、组合条件、禁用态/空状态/错误态、兼容性/稳定性、性能影响、审计/日志/通知、副作用和可恢复路径；不能只写一个笼统用例替代多个可验证方向。
- 如果产品版图缺失但 `docs/test-assets/modules/` 中存在相关归档文件，应读取归档文件，并登记产品版图待同步问题。
- 不得仅凭 AI 对话记忆判断已有模块能力、已有用例、可复用测试数据或跨模块依赖。
- 当前模块依赖已有模块能力时，优先引用已有用例 ID 作为前置条件，不重复复制已有用例。
- 跨模块用例只保留一个主归属模块，依赖模块通过产品模块地图、业务对象地图、业务链路地图、用例资产索引、跨模块依赖关系、标签、备注和关联用例 ID 追踪。
- 生成、评审或人工修订后的最终版测试设计应回存 `docs/test-assets/modules/`；测试系统导入文件副本应回存 `docs/test-assets/imports/`；客户交付件应保存到 `docs/test-design/current/` 或 `docs/test-design/deliverables/`；产品模块、业务对象、业务链路、页面元素、用例资产、模块能力、跨模块依赖、可复用测试数据、变更影响分析和变更记录应同步到 `product-map.xlsx`。
- 生成正式测试设计 Excel 后，应运行 `scripts/validate-test-design-deliverable.ps1 -WorkbookPath <测试设计.xlsx>`；大范围任务追加 `-BatchStatusPath <batch-status.csv>`；如果 `batch-status.csv` 同级存在 `page-discovery.csv`，脚本会自动使用 `docs/test-assets/product-map.xlsx` 启用产品版图同步校验；也可以显式追加 `-ProductMapPath docs/test-assets/product-map.xlsx -PageDiscoveryPath <page-discovery.csv>`，校验页面实探、正式 Excel 和产品版图之间的页面元素、关联用例、用例资产索引和变更记录是否同步。

## 页面材料处理规则

如果输入包含需求文档、截图、原型、页面 URL、菜单路径或“页面所有可点击功能”等要求，必须先识别页面元素，再生成用例。

材料使用优先级：

1. 需求、用户故事、验收标准和业务规则。
2. 用户提供的文档、截图、原型图、操作说明。
3. 可访问页面的实际表现，包括 DOM、可见文案、菜单、按钮、链接、图标按钮、下拉项、开关、输入框、弹窗和表格行操作。
4. 通过浏览器或 computer use 观察到的页面行为。
5. 无法确认的信息必须记录为待确认问题。

如果文档或截图不足以确认所有可点击元素，并且存在可访问页面，应使用浏览器能力查看页面；如果只能通过桌面窗口、远程会话或既有登录态访问，可使用 computer use 辅助观察。

## 页面实探数据安全规则

使用浏览器或 computer use 观察页面时，必须遵守：

- 不得保存、提交、删除、审批、驳回、禁用、启用、发布、下线、分配、转移、批量处理或改变任何页面已有数据的内容或状态。
- 可以新增带有明显测试标识的数据，例如名称、备注或描述中包含 `AI_TEST`、`CODEX_TEST`、当前日期或任务编号。
- 只有确认是本次实探自己创建的数据，才允许继续编辑、删除、状态变更或清理。
- 如果无法判断某条数据是否由自己创建，必须按已有数据处理，只允许查看、搜索、筛选、排序、分页、打开详情，或进入编辑/修改页面观察字段、校验、联动、多级子菜单和保存前提示；不得点击保存、提交、确认修改等会落库或改变状态的动作。
- 对本次实探自己创建且带有测试标识的数据，可以执行保存、删除、审批、驳回、禁用、启用、发布、下线、分配、转移、批量处理、外部调用等敏感或高风险操作，以获取真实提示、状态变化和数据影响；最终用例必须明确前置条件为“仅操作本次创建的测试数据”。对已有数据仍不得执行这些动作。
- 测试用例可以设计针对已有数据的修改/删除/状态变更场景，但实际页面实探不得对已有数据执行这些操作。

## 用户提供测试数据规则

如果用户提供测试数据，必须遵守：

- 优先用用户提供的测试数据走通主流程、关键分支、字段校验、搜索筛选、详情查看、编辑页字段联动等页面功能。
- 对明确标注为异常、边界、非法或空值的数据，应观察页面真实错误提示、禁用态、校验位置、接口/页面响应和可恢复路径，并据此补充负向用例。
- 如果用户提供的数据明确用于本次测试并带有测试标识，可以用于新增、保存、编辑、删除、状态变更等会落库的操作；如果数据会修改已有数据或无法确认数据归属，只能用于查看、填入后观察校验、记录预期或设计用例，不得保存或提交。
- 如果用户提供的数据包含真实密码、Token、密钥、生产账号、手机号、身份证号等敏感信息，最终测试用例必须使用占位符或脱敏值，不得原样输出。
- 页面实探得到的实际提示文案、字段限制、联动效果、跳转结果、状态变化和错误信息，必须体现在 `操作步骤` 与 `预期结果` 中。

## 业务逻辑与页面交互融合规则

设计文档、需求资料和页面信息经常各自不完整，生成测试用例时必须融合使用：

- 从设计文档、需求说明、接口文档、验收标准中抽取业务目标、角色权限、状态流转、字段规则、异常规则、数据影响、通知/日志/审计等业务逻辑。
- 从页面截图、原型或实际页面中抽取具体入口、控件名称、点击路径、表单字段、弹窗按钮、表格行操作、筛选/排序/分页、禁用态、空状态和错误态。
- 如果文档只描述大概功能而页面展示了具体入口，应把页面入口纳入覆盖清单，并用文档业务逻辑补足测试场景。
- 如果页面存在文档未提到的功能入口，不得直接假设完整业务规则；应生成基础交互用例，并把业务规则、权限、数据影响或验收标准登记为待确认问题。
- 如果文档描述的业务规则在页面没有明显入口，应记录入口缺失或实现路径不明，并根据可访问页面继续寻找可能入口。
- 功能测试用例必须同时体现业务意图和页面操作，禁止只按按钮机械生成用例，也禁止只复述文档而缺少具体页面步骤。页面类用例的 `操作步骤` 必须包含如何进入目标功能：登录/打开系统、进入一级菜单、二级/三级菜单或目标页面，再写具体控件操作。

## Excel 输出规则

正式 Excel 至少包含以下 Sheet：

- `测试设计总览`
- `需求用户故事拆解`
- `测试场景矩阵`
- `功能测试用例`
- `性能测试设计`
- `风险与待确认问题`
- `自动化建议`
- `页面元素覆盖清单`（页面/截图/原型/可访问系统场景必须包含；纯接口或纯后端需求可为空并说明不适用）

`页面元素覆盖清单` 只用于记录页面元素、业务依据、覆盖状态、发现方式、素材来源和关联的 `覆盖用例 ID`。不得在该 Sheet 中编写独立测试用例、操作步骤、测试数据或完整预期结果正文。

测试系统导入内容不再写入正式测试设计工作簿；需要导入时，应生成基于 `docs/test-design/测试用例模板.xlsx` 副本的独立导入文件，其中测试步骤、预期结果、前置条件、名称、类型、级别和标签必须从 `功能测试用例` Sheet 映射得到。

`执行方式` 表示测试系统中该条用例当前如何执行，默认填写 `手动`。`是否适合自动化`、`自动化建议`、AI 页面实探或工具辅助点击只说明自动化建设价值或生成过程，不代表该用例已经具备自动化执行能力，不得因此填写 `自动化`。

Excel 单元格要求：

- `前置条件`、`操作步骤`、`预期结果` 三列必须启用自动换行。
- 表头必须冻结。
- 字段命名必须统一。
- 优先级、状态、是否适合自动化等字段应使用固定枚举值。
- `功能测试用例.用例标题` 和测试系统导入文件中的 `测试用例名称` 必须使用正式、简洁、可检索的测试项表达，避免口语化、聊天式或操作随笔式标题。
- 由于测试系统导入后没有独立 `功能点` 字段，`用例标题` 必须使用 `功能点-当前用例标题` 格式；短横线前后不加空格，例如 `搜索筛选-按智能体名称查询列表`。
- 短横线前的功能点必须与本行 `功能点` 字段一致或为其清晰归纳，不得省略；短横线后的当前用例标题建议采用“验证/检查/按/通过 + 对象/条件 + 行为/结果”的结构。
- 标题长度宜控制在 8 到 40 个中文字符；同一小功能块标题风格保持一致，能从标题区分主流程、异常、边界、权限/状态或数据一致性。
- 正例：`搜索筛选-按智能体名称查询列表`、`表单校验-服务地址为空时禁止测试Agent`、`删除管理-删除AI_TEST智能体后列表刷新`、`编辑管理-编辑模式下接入类型不可修改`。
- 反例：`搜索一下天气看看`、`测试Agent能不能成功`、`点删除是否OK`、`打开页面看一下`。

`页面元素覆盖清单` 必须包含以下字段：

- 元素 ID
- Story ID/需求 ID
- 页面/入口
- 页面 URL/菜单路径
- 元素名称/文案
- 元素类型
- 交互方式
- 前置状态/权限
- 预期行为
- 业务依据/规则来源
- 覆盖用例 ID
- 覆盖状态
- 发现方式
- 素材来源
- 待确认问题/备注

`覆盖状态` 固定枚举值：

- 已覆盖
- 不适用
- 不测范围
- 待确认

## 用例数量规则

如果输入是敏捷用户故事：

- 每个 Story 至少 10 条功能测试用例。
- 如果 Story 很小，也必须从边界、异常、权限、数据、幂等、状态、日志、副作用、兼容性和性能角度补足覆盖。
- 不允许通过同质化用例凑数量。

## 用例编排规则

- 功能测试用例必须按模块、页面、业务流程和小功能块组织。
- 同一小功能块的用例必须连续放在附近，例如新增、查看、编辑、删除、搜索、筛选、分页、导入、导出、审批等分别成组。
- 每个小功能块内部再按主流程、异常流程、边界值、权限/状态、数据一致性、兼容性或稳定性排序。
- 不要把同一小功能块的用例分散到表格不同位置；如需跨功能关联，应在备注或关联风险中说明。

## 测试系统导入规则

独立测试系统导入文件字段顺序必须为：

1. 一级模块系统编号
2. 一级模块名称
3. 二级模块系统编号
4. 二级模块名称
5. 三级模块系统编号
6. 三级模块名称
7. 四级模块系统编号
8. 四级模块名称
9. 五级模块系统编号
10. 五级模块名称
11. 其他模块系统编号
12. 其他模块名称
13. 测试用例系统编号
14. 测试用例序号
15. 测试用例名称
16. 测试步骤描述
17. 测试步骤预期结果
18. 测试类型
19. 测试用例级别
20. 执行方式
21. 测试用例说明
22. 前置条件
23. 维护人
24. 标签
25. 备注
26. 作者

映射要求：

- `测试用例名称` = `功能测试用例.用例标题`，必须保持正式、简洁、可检索，不得口语化或写成操作随笔；导入值必须保留 `功能点-当前用例标题` 格式。
- 处理历史用例或外部输入时，如果 `功能测试用例.用例标题` 尚未带功能点前缀，生成独立导入文件前必须先按本行 `功能点` 字段规范化为 `功能点-当前用例标题`。
- `测试步骤描述` = `功能测试用例.操作步骤`。
- `测试步骤预期结果` = `功能测试用例.预期结果`。
- `前置条件` = `功能测试用例.前置条件`。
- `测试类型` 必须使用测试系统枚举：功能测试、性能规格测试、可靠性测试、兼容性测试、可维护性测试、安全性测试、易用性测试。
- `测试用例级别` 必须使用测试系统枚举：L1、L2、L3、L4。
- `执行方式` 必须使用测试系统下拉枚举，只能填写 `自动化` 或 `手动`。
- `执行方式` 默认填写 `手动`；只有已有可运行、可维护并覆盖该用例主要校验点的自动化资产，且本次交付明确按自动化导入或关联自动化资产时，才填写 `自动化`。
- 模块名称列应优先按业务模块层级拆分；没有系统编号时，系统编号列可以留空。
- 测试系统自动生成字段不强行填写，例如测试用例系统编号。
- 测试系统自动生成字段不需要填写，不得编造值；包括 `测试用例系统编号`、`评审状态`、`自动化状态`、`执行次数`、`创建时间`、`作者`、`最后修改时间`、`用例来源` 等。
- 测试系统模板中的红色字段和必填字段必须按每条测试用例动态生成，不得固定写示例值或通用占位值。
- 红色字段包括 `测试用例名称`、`测试步骤描述`、`测试步骤预期结果`、`测试类型`、`测试用例级别`、`执行方式`。
- 必填字段至少包括 `测试用例名称`、`测试类型`、`测试用例级别`、`执行方式`；如后续测试系统模板或字段解释发生变化，应以最新模板和字段解释为准。
- 下拉框字段只能选择测试系统模板中的下拉内容，不得自由发挥；当前下拉字段包括 `测试类型`、`测试用例级别`、`执行方式`。
- 输出独立导入 `.xlsx` 时必须保留模板中的 Excel 数据验证下拉框、必填字段样式、标红字段和自动生成字段空值；不得通过新建空白 Sheet 或仿制表头替代模板副本。
- `测试用例序号` 应根据模块层级和小功能块顺序动态生成，支持层级化序号，例如 `1`、`1.1`、`1.2`。
- `测试用例说明`、`标签`、`备注`、模块名称等字段也应结合用例内容动态生成，不得整列重复无意义文本。

## 编号换行规则

以下字段必须使用编号换行：

- `前置条件`
- `操作步骤`
- `预期结果`

格式必须为：

```text
1. xxx
2. xxx
3. xxx
```

如果只有一项，也必须写成：

```text
1. xxx
```

禁止写法：

- 一整段长文本
- 用分号、顿号、逗号堆叠
- 使用 `-`、`*` 或无编号列表
- 只写 `验证正常`、`结果正确`、`符合预期`

## 覆盖维度规则

每个功能点至少评估以下维度：

- 正常流程
- 异常流程
- 边界值
- 输入校验
- 权限控制
- 角色差异
- 状态流转
- 幂等性和重复提交
- 数据一致性
- 并发或竞争条件
- 兼容性
- 性能和稳定性
- 日志、通知、审计、消息等副作用
- 安全风险

如果存在页面、截图、原型或可访问系统，还必须评估以下维度：

- 页面入口和导航路径
- 所有按钮、链接、菜单、Tab、图标按钮、表格行操作和批量操作
- 搜索、筛选、排序、分页、重置、导入、导出、上传、下载、预览
- 弹窗、抽屉、确认框、取消、关闭和返回
- 输入框、下拉框、日期选择器、开关、复选框、单选框
- 禁用态、隐藏态、空状态、加载态、错误态、无权限态
- 不同角色、不同数据状态下元素可见性和可用性

## 优先级规则

- P0：核心链路、资金、订单、权限、数据安全、不可恢复损失。
- P1：主要功能、常见异常、重要边界、关键回归范围。
- P2：低频场景、兼容性、提示文案、非核心分支。
- P3：体验优化、弱影响展示、边缘场景。

## 性能测试规则

`性能测试设计` Sheet 不允许为空。

必须包含：

- 关键业务链路
- 测试类型
- 并发或用户量假设
- TPS/QPS 或吞吐目标
- 响应时间目标
- 数据量级
- 测试时长
- 监控指标
- 通过标准
- 造数策略
- 风险说明

如果不纳入本轮测试，必须说明原因，并给出回归监控建议。

## 可执行性规则

每条测试用例必须满足：

- 有明确前置条件。
- 有明确测试数据，或说明测试数据准备方式。
- 如果用户提供测试数据，应优先引用这些数据或其脱敏占位符，并说明数据用途，例如正常数据、边界数据、非法格式、重复数据、无权限数据、缺失必填项等。
- 操作步骤具体。
- 页面交互类用例必须写明具体页面元素、控件名称或入口文案。
- 页面交互类用例的预期结果必须体现业务规则来源，例如状态变化、数据展示/落库、权限拦截、接口结果、通知、日志、审计或验收标准。
- 预期结果可观察、可验证。
- 一条用例只验证一个核心场景。
- 不把多个独立场景混在同一条用例中。

## 追踪规则

每条测试场景和测试用例都必须关联：

- Story ID 或需求 ID
- 功能点
- 测试维度
- 关联风险或验收标准

如输入没有明确 ID，应自动生成临时 ID，例如 `STORY-001`、`REQ-001`。

## 页面元素覆盖规则

每个识别到的可点击/可交互元素至少满足以下之一：

- 在功能测试用例中有关联用例 ID。
- 标记为仅展示/不可操作，并说明依据。
- 标记为不测范围，并说明原因。
- 登记为待确认问题。

禁止只按页面大功能粗略覆盖，遗漏行内按钮、图标按钮、二级菜单、更多操作、禁用按钮、弹窗按钮、分页、排序、筛选、上传、下载等入口。

页面元素覆盖清单中的 `覆盖用例 ID` 必须指向 `功能测试用例` Sheet 中的用例，性能相关元素或链路应通过 `性能测试设计` Sheet 中的性能场景 ID 体现，不得在覆盖清单中替代正式测试用例。

## 外网到内网升级规则

- 普通框架升级以脚本升级为主、手动确认兜底；外网使用 `scripts/new-framework-upgrade-package.ps1` 生成升级包，内网使用 `scripts/upgrade-framework.ps1 -PackagePath <升级包>` 应用升级包。
- `docs/test-assets/`、`docs/test-design/current/`、`docs/test-design/deliverables/` 是受保护目录，普通框架升级不得覆盖、删除或清空。标识：PROTECTED_ASSET_DIRS。
- `VERSION` 中的 `framework_version` 表示框架版本，`asset_schema_version` 表示内部资产结构版本。
- `product-map.xlsx` 是主要可能演进的内部资产结构；历史归档 Excel 默认作为历史快照保留，不因普通框架升级而批量重写。
- `asset_schema_version` 变化或产品版图结构变化时，必须通过升级清单、校验脚本和迁移脚本处理；迁移脚本只能读取旧资产并增量补齐，不能用空模板覆盖内网真实资产。
- 分批必须按当前产品或模块可识别的最深标题级别执行，例如一级标题、二级标题、三级标题、四级标题等，哪个标题级别最小就以哪个最小标题作为一个批次。每个已通过批次只能覆盖 1 个最小标题路径，`最小标题路径` 使用 `一级>二级>三级>四级` 形式记录唯一叶子节点；禁止合并多个最小标题，禁止再拆分一个最小标题为多个批次。已通过批次的导入文件路径必须真实存在，并能与归档测试设计逐个匹配校验。
- 当任务范围是全产品、多个一级模块或大模块时，必须先遍历一级菜单、二级菜单、三级菜单及更深层标题，拿到菜单轮廓、页面清单和功能地图后输出分批设计计划，不得一次性生成完整测试用例；各批按最小标题路径执行，所有批次完成后再做跨模块汇总。
- 当任务范围超过一个最小标题时，必须建立批次队列并按最小标题路径逐批执行。
