# DFX 12 维度测试策略矩阵

本文件用于约束异常值、边界值、性能、安全、兼容、可靠性等测试策略的生成方式，避免只写一句笼统策略。每个批次、每个小功能块、每个可交互页面元素都应基于适用的 DFX 维度选择具体场景，并落到正式测试设计的测试点、测试数据、操作步骤和预期结果中。

## 生成规则

- DFX 是扩展检查矩阵，不是用例生成主轴；禁止按“每个 DFX 场景生成一条用例”的方式压缩页面功能。
- 正确生成顺序必须是：页面/模块 -> 功能点 -> 页面元素/交互路径 -> 数据状态/业务状态 -> 适用 DFX 维度 -> 测试用例。
- 每个小功能块必须先建立元素覆盖骨架，再按 DFX 扩展；DFX 只能补充正向、边界、异常、逆向、权限、状态、数据一致、可用性、可靠性等测试方向，不能替代按钮、输入框、下拉框、分页、弹窗、表格行操作和 CRUD 路径的逐项覆盖。
- DFX 扩展结果必须写入 `element-case-plan.csv` 的 `适用DFX维度`、`适用DFX场景`、`测试设计方向`、`应生成用例数` 和 `计划用例ID` 字段；正式功能用例的 `实际用例ID` 必须回填到计划中。
- 模块或批次正式写测试用例前，必须先做 DFX 覆盖评估：结合产品理解摘要、页面实探、需求/设计/接口文档、历史归档、测试数据和风险项，逐项判断 12 个维度及 4 类场景是否适用于当前模块。
- DFX 覆盖评估必须输出适用、不适用、待确认和需补充证据的结论；适用项进入功能测试用例、性能测试设计或自动化建议，不适用项说明原因，待确认项进入风险与待确认问题并在用户确认后动态调整测试范围。
- DFX 覆盖评估不能只停留在 `dfx-assessment.md` 或过程说明中；正式测试设计必须在 `测试场景矩阵`、`功能测试用例`、`性能测试设计`、`风险与待确认问题` 和 `页面元素覆盖清单` 中填写对应的 `DFX维度` 与 `DFX场景` 字段。
- `场景类型`、`正向/反向` 已废弃，不能继续作为测试策略字段；正向、反向、异常、边界等策略统一通过 `DFX场景` 表达。
- 不得只写“异常值、边界值、性能测试需覆盖”等一句话；必须说明适用维度、具体场景、触发数据、操作路径、观察点、预期结果和恢复路径。
- 每个小功能块默认至少覆盖 DFT 功能、DFB 业务、DFS 安全或权限、DFR 可靠或数据一致、DFU 可用性；有接口、性能、部署、运维、兼容特征时追加对应维度。
- 页面元素类用例必须把 DFT 的正向流程、边界值、异常输入、逆向操作转化为可执行步骤，例如输入最大长度、空值、重复值、非法格式、取消/关闭/返回、重复提交和数据不变。
- 性能相关场景必须写入 `性能测试设计` Sheet，不得只放在功能测试用例备注中；性能场景应明确目标用户量/并发数、响应时间目标、数据量级、监控指标、通过标准和造数策略。
- `功能测试用例` Sheet 禁止填写 `测试类型=性能规格测试`，禁止使用 `DFX维度=DFP性能`；`DFX极端` 中的压力极限、资源耗尽、并发极限默认进入性能测试设计、风险或自动化建议，不进入功能测试用例。
- 接口、超时、重试、幂等、数据一致、日志、告警、监控、回滚等系统性场景如果当前页面无法直接执行，应写入风险与待确认问题、性能测试设计或自动化建议，并说明需要的接口、日志、监控或环境支持。
- 破坏性、资源耗尽、并发极限等 DFX 极端场景不得在共享或生产样数据上直接执行；仅允许在隔离测试环境、压测环境或本次创建的测试数据上设计，并明确安全边界、回滚或清理策略。

## 用例密度预算

- 每个最小批次在生成用例前必须根据页面元素覆盖骨架估算最低用例量，防止 DFX 接入后压缩用例；该预算必须落入 `element-case-plan.csv` 的 `应生成用例数`，不得由模型随意统一填写为 1。
- 最低预算必须结合“元素类型/交互方式 + 适用DFX维度/适用DFX场景”计算：输入框至少覆盖正常值、空值、非法值、边界值；选择类控件至少覆盖展开、代表项选择、联动/无联动、清空/重置；分页至少覆盖条数切换、上一页、下一页、页码跳转、边界页和筛选后重置；新增/编辑/删除必须分别覆盖成功、失败/校验、取消、保存后校验和恢复路径。
- 涉及 DFS 安全、DFR 可靠、DFU 可用、DFB 业务、配置生效、权限、密钥、状态流转、数据一致性或 CRUD 闭环时，必须在基础元素用例上追加对应 DFX 扩展用例；DFP 性能和 DFX 极端中不适合手工功能执行的场景写入性能测试设计、风险或自动化建议。
- 写功能用例前必须运行阶段性门禁：`powershell -ExecutionPolicy Bypass -File scripts/run-test-design.ps1 validate-batch-artifacts --run-dir <batch-run-dir> --phase plan`。门禁会校验 `page-discovery.csv`、`element-case-plan.csv`、`test-data-lifecycle.csv` 的表头、列数、元素映射、DFX 最低用例数和 CRUD 生命周期。
- 实际功能测试用例数量明显低于已生成页面元素数量时，必须补充用例或说明不可测原因；不得用一条 DFX 场景概括多个可点击、可输入、可选择元素。
- 如果 `element-case-plan.csv` 中某个元素的 DFX 计划未生成用例，必须填写 `未生成原因`，并同步到风险与待确认问题；禁止仅通过降低应生成用例数来规避覆盖。

## 用例落地格式

生成功能测试用例时，每条用例必须填写 `DFX维度` 和 `DFX场景`，并在 `测试数据`、`操作步骤`、`预期结果` 至少体现：

1. 维度与场景：例如 `DFT-边界值`、`DFS-权限控制`、`DFR-幂等性`。
2. 输入或状态：正常值、边界值、非法值、重复值、无权限角色、异常状态、超时/重试条件、并发条件等。
3. 观察点：页面提示、字段校验位置、按钮禁用态、数据落库、列表刷新、状态流转、日志/审计、通知、接口错误码、监控指标。
4. 预期结果：成功、失败、拦截、数据不变、可恢复、可重试、回滚成功、告警触发或指标达标。

## 批次自检

每批完成前，应检查：

- 功能测试用例是否至少覆盖当前小功能块适用的 DFT、DFB、DFS、DFR、DFU 场景。
- 性能测试设计是否覆盖 DFP 响应时间、并发处理、大数据量和资源监控中适用场景。
- 风险与待确认问题是否记录当前无法验证但对产品质量重要的 DFI、DFD、DFO、DFX 场景。
- 页面元素覆盖清单中的输入、选择、按钮、弹窗、表格行操作、批量操作、分页、筛选、排序等元素是否都映射到可执行 DFX 场景或明确标注不适用原因。
