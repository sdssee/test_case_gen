# DFX驱动测试策略

DFX用于理解功能、指导实探和扩展Case，不是在用例写完后机械匹配标签，也不要求每个功能展开固定矩阵。

## 策略维度

| 维度 | 典型方向 |
| --- | --- |
| DFT 功能 | 正向流程、有效等价类、空值、格式异常、边界、逆向操作 |
| DFP 性能 | 响应时间、并发、大数据量、资源指标 |
| DFI 接口 | 参数、协议、错误码、超时重试 |
| DFC 兼容 | 浏览器、操作系统、屏幕、数据格式 |
| DFS 安全 | 认证、权限、会话、注入和产品自身的数据展示策略 |
| DFR 可靠 | 故障恢复、数据一致、幂等、断点续传 |
| DFM 维护 | 配置生效、日志追踪、灰度和回滚 |
| DFU 可用 | 操作反馈、错误提示、用户引导、可达性 |
| DFD 部署 | 安装、升级、卸载、配置迁移 |
| DFO 运维 | 监控、告警、自愈、容量 |
| DFB 业务 | 业务流程、数据准确、端到端、统计 |
| DFX 极端 | 压力极限、资源耗尽、并发极限、破坏性条件 |

## 与实探配合

- 扫描页面时先识别控件、业务对象、状态和依赖，再选择适用DFX方向；清单中不存在的新能力动态加入。
- 页面可安全验证的适用场景在连续实探中直接执行，并记录为`已实测`；只观察未触发的记为`页面观察`。
- 页面无法执行但基于功能有价值的扩展记为`DFX设计`；只有页面、文档和现有数据均无法确认且影响设计的内容记为`待确认`。
- 每个有限选项和每个实测有效输入等价类分别形成baseline Case。必填空值、明确格式异常、边界、权限、可靠性等按适用性扩展，不替代baseline。
- 新增、编辑和配置项的每个分支必须在本轮测试数据上实际保存并验证回显或效果；多个配置项采用单因素，不默认组合。

## Case落地

- 适用DFX必须在Case规划时直接进入标题、测试点、数据、步骤和具体预期，不等待最终Review补齐。
- 功能、业务、安全、可靠、易用、兼容和接口方向按适用性进入功能Case；不适用方向不生成占位用例。
- 性能方向进入`性能测试设计`。有需求阈值时引用需求；无阈值时写实测基线、建议目标或待确认，不能编造成正式验收标准。
- 页面无法直接验证的接口、日志、监控、部署和极端方向进入风险、性能设计或自动化建议，并注明所需环境与观察点。
- 破坏性、资源耗尽或并发极限只在隔离环境设计或执行；取消内容脱敏不扩大操作授权。

## 轻量自检

每个JSON分片写入时只检查：适用DFX是否已经落到具体数据、步骤和预期；baseline是否仍独立存在；性能场景是否进入性能Sheet。最终只做一次跨产物语义Review，不逐维度拒绝、不自动补齐、不启动返工循环。
