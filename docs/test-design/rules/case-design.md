# 测试计划与用例规则

## 独立功能与计划

从 facts 动态识别具有独立触发和可观察结果的用户功能，不依赖预设功能名称或固定控件清单。页面上实际出现的查询、状态切换、内容浏览、详情、数据维护等能力均按同一规则判断；单个控件、具体选项和普通字段本身不自动等同于独立功能。

运行时从facts生成只读计划骨架：除空值、边界、有限选项等元素级提示外，直接带入实测动作、输入类/选项、稳定结果锚点和已观察状态。模型据此为每个功能一次填写 `design_context`（用户目标、角色、业务价值、验收标准、业务规则、依赖、后置影响、真实依据）和 `automation_profile`，补充Case意图，并在唯一 `check_assignments` 中分配检查点。性能/风险适用时结构化记录，不适用时给出真实原因。系统从该账本派生主验证和辅助使用覆盖，不维护独立DFX账本。

一个页面事务可以连续实探多个选项或输入类别，但 `required_case_branches` 中每个有限选项和每个实测有效输入等价类必须分别绑定一个独立 baseline Case；不同分支不得共用同一 Case。一个 Case 可以辅助使用多个控件，但不能以此替代其他控件所属功能的独立覆盖，也不默认展开跨维度组合。

## 用例正文

- 只按 `case-plan.json` 的稳定 Case ID 和功能顺序写作，不临时发明功能、场景和预期。
- 计划标题只写具体场景，不重复功能名；最终标题由系统按“功能点-具体场景”形成且不得为空。
- 每条用例第一步使用完整实际菜单路径，例如“进入告警管理-告警列表”，不逐级拆分菜单。
- 前置条件只描述角色、环境、状态和数据；菜单导航写入步骤。
- 使用具体且脱敏的数据，不写“输入测试数据”等占位描述。
- 模型为每个业务步骤写 `{action, expected}`；动作必须包含事实检查中的完整输入、具体选项和触发过程。系统按计划分配顺序自动注入 `source_check`。一个步骤对应一个检查点，预期优先满足结果锚点中的 `stable_tokens`，不得把偶发计数、进度等样本值固化为稳定结论。导航步骤不绑定检查点，内部来源不导出Excel。
- 每条用例填写 `automation_value`、`automation_priority`；只有与功能默认画像不同才填写 `automation_override`，不复制整套自动化说明。
- 同功能用例连续排列。公共导航允许重复，核心操作、数据和预期不得整段重复。
- 计划和用例均在各自单文件中按功能 upsert；再次提交同一功能时替换该功能块，其他功能保持不变，精确重复提交不产生重复内容。
- 禁止截图要求、UID、DOM、选择器、事实编号和页面工具术语。

CRUD、编辑和配置用例必须写出提交后的查询、重开回显、实际效果和恢复/清理。无效分支写清具体拦截与数据未变化，不能编造成成功结果。
