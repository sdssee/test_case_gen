# 用例设计规则

## 设计输入

- 页面实探是已实现功能的主要事实来源；需求、用户故事、设计文档、接口资料和模型推断用于辅助理解。
- 正式设计前完成DFX适用性评估。DFX在规划Case时驱动扩展，不在最终Review阶段机械补齐。
- 事实来源必须区分`已实测`、`页面观察`、`DFX设计`和`待确认`；设计场景不能冒充实测结论。

## baseline Case

- 每个有限选项分别形成一个baseline Case。
- 每个有效输入等价类分别形成一个baseline Case。
- 验证一个选项时其他控件使用默认稳定值；验证一种输入时选择控件使用默认选项。
- 不默认生成选项与输入的笛卡尔积；只有页面明确存在依赖、互斥、联动或不同业务结果时才增加必要组合。
- 元素在其他Case中被辅助使用，不能替代其独立主验证Case；DFX扩展不能替代baseline Case。

## CRUD与可选配置

- 新增必须使用本轮测试数据实际保存，并验证创建成功提示、列表/详情持久化结果及配置实际效果。
- 编辑必须保存成功，重新进入详情或编辑页验证回显、生效及其他字段未变化。
- 删除只作用于本轮测试数据，分别验证取消后数据仍存在、确认后删除成功或被明确拦截。
- 每个可选配置至少验证不配置和配置。多个配置项采用“全部默认、只配置A、只配置B……”的单因素方式，不默认展开组合矩阵。
- 有限选项位于新增或编辑事务中时，每个选项Case都必须完成对应的创建或保存验证。

## 编排与标题

- 用例按模块、页面、功能点稳定排序；同一小功能块的baseline和DFX扩展Case连续排列。
- 标题使用`功能点-当前用例标题`，短横线前后不加空格。
- 标题、测试点、测试数据、操作步骤和预期结果必须围绕同一验证目标。

## 步骤与预期

- 前置条件、操作步骤、预期结果使用`1. 2. 3.`连续编号并换行。
- 环境准备写入前置条件；操作步骤第一步从系统入口按本轮真实菜单层级导航到目标页面，不得照抄规则示意名称或保留占位符。
- 步骤写明当前功能、选项、输入、配置、触发动作和观察路径。
- 预期写明当前字段提示、选中/联动状态、保存/回显结果、列表/详情变化、数据是否保持或删除等可观察结果。
- 禁止单独使用“页面正常响应”“系统处理正确”“结果符合预期”“确认操作完成后页面功能正常可用”等通用填充语句。

## 按操作类型判断业务闭环

- 下拉/单选/开关：实际选择并观察选中、联动或业务结果；不要求切回原值。
- 输入校验：实际输入、触发校验并记录具体提示或结果；不要求固定清空恢复。
- 查询/搜索/诊断/刷新：获得并验证结果或错误反馈即完成；重置是独立功能时保留独立Case。
- 弹窗/抽屉：按当前Case目标分别验证打开、确认、取消或关闭，不要求一条Case覆盖全部动作。
- 新增/编辑/删除/配置变更：必须达到本节CRUD定义的业务结果。
- 只有改变共享环境、全局配置或后续Case状态时才要求恢复。

## 生成前轻量自检

每个既有JSON分片写入前一次完成：

1. 自动连续编号。
2. 完整导航。
3. 功能内稳定排序。
4. 标题、步骤和预期一致。
5. 具体选项、输入和配置进入步骤与预期。
6. 检查用例ID、标题、步骤与预期的确定性重复。

不做模糊相似度拒绝，不生成临时修复脚本，不改变既有多JSON分片设计。

## 性能与DFX

- 适用的功能、业务、安全、可靠、易用、兼容、接口等DFX场景落实到Case。
- 性能场景写入`性能测试设计`；阈值必须标明需求明确、实测基线、建议目标或待确认，不能编造为正式验收标准。
- 当前页面无法验证的接口、日志、监控、部署和极端场景进入风险、性能设计或自动化建议，并说明所需支持。
