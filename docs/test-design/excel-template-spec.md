## Sheet 1：测试设计总览

字段：

- 项目/模块
- 需求名称
- 版本/迭代
- 测试负责人
- 需求来源
- 测试范围
- 不测范围
- 测试类型
- 测试环境
- 主要风险
- 准入条件
- 准出条件
- 待确认问题

表头位于第1行，本轮总览作为一条横向记录写入第2行；不得转换为纵向键值表。

字段：

- Story ID/需求 ID
- 用户故事/需求描述
- 角色
- 业务价值
- 验收标准
- 业务规则
- 前置条件
- 后置影响
- 依赖系统

## Sheet 3：测试场景矩阵

字段：

- 场景 ID
- Story ID/需求 ID
- 功能点
- 测试维度
- DFX维度
- DFX场景
- 测试对象/页面元素
- 输入数据/状态条件
- 观察点
- 风险等级
- 优先级
- 是否生成用例
- 备注

字段约束：

- `场景类型` 和 `正向/反向` 已废弃，不再作为测试策略字段；测试策略统一由 `DFX维度` 和 `DFX场景` 表达。
- `测试维度` 保留为测试类型大类，例如功能测试、性能规格测试、安全性测试、可靠性测试、兼容性测试、可维护性测试、易用性测试。
- `DFX维度` 只能使用 `DFT功能`、`DFP性能`、`DFI接口`、`DFC兼容`、`DFS安全`、`DFR可靠`、`DFM维护`、`DFU可用`、`DFD部署`、`DFO运维`、`DFB业务`、`DFX极端`。
- `DFX场景` 必须属于对应 DFX 维度的 4 个场景；例如 `DFT功能` 只能对应正向流程、边界值、异常输入、逆向操作。
- `测试对象/页面元素`、`输入数据/状态条件`、`观察点` 必须把 DFX 场景转化为可执行、可验证的测试设计输入，不能只写笼统策略。

## Sheet 4：功能测试用例

字段：

- 用例 ID
- Story ID/需求 ID
- 模块
- 功能点
- 用例标题
- 优先级
- 测试类型
- DFX维度
- DFX场景
- 前置条件
- 测试数据
- 操作步骤
- 预期结果
- 实际结果
- 执行状态
- 是否适合自动化
- 关联风险
- 备注

编排规则：

- `用例标题` 和测试系统导入文件中的 `测试用例名称` 必须使用正式、简洁、可检索的测试项表达，避免口语化、聊天式或操作随笔式标题。
- 由于测试系统导入后没有独立 `功能点` 字段，`用例标题` 必须使用 `功能点-当前用例标题` 格式；短横线前后不加空格，例如 `搜索筛选-按智能体名称查询列表`。
- 短横线前的功能点必须与本行 `功能点` 字段一致或为其清晰归纳，不得省略；短横线后的当前用例标题建议采用“验证/检查/按/通过 + 对象/条件 + 行为/结果”的结构，优先体现测试点和业务对象。
- 标题长度宜控制在 8 到 40 个中文字符；同一小功能块标题风格保持一致，能从标题区分主流程、异常、边界、权限/状态或数据一致性。
- 正例：`搜索筛选-按智能体名称查询列表`、`表单校验-服务地址为空时禁止测试Agent`、`删除管理-删除AI_TEST智能体后列表刷新`、`编辑管理-编辑模式下接入类型不可修改`。
- 反例：`搜索一下天气看看`、`测试Agent能不能成功`、`点删除是否OK`、`打开页面看一下`。
- 用例必须按模块、页面、业务流程和小功能块组织。
- 同一小功能块的用例必须连续放在附近，例如新增、查看、编辑、删除、搜索、筛选、分页、导入、导出、审批等分别成组。
- 每条功能测试用例都必须填写`DFX维度`和`DFX场景`，并把场景落到测试数据、操作步骤和具体预期中；只有改变共享状态时才要求恢复路径。
- 每个小功能块内部按baseline Case、适用DFX场景和业务流程连续编排；不适用方向不机械展开，不默认做组合积。
- 不要把同一小功能块的用例分散到表格不同位置；如需跨功能关联，应在备注或关联风险中说明。

测试数据规则：

- 如果用户提供测试数据，应优先写入或引用到 `测试数据` 字段。
- 正常数据用于确认成功路径、跳转结果、状态变化和页面展示。
- 异常、边界、非法、空值、重复、无权限等数据用于补充负向用例，并记录页面真实提示、校验位置和恢复路径。
- 用户允许在内网流通的测试数据可原样写入Excel；框架不对IP、URL、账号、密码、Token、密钥、Cookie、部署路径和测试载荷做敏感内容校验或占位符替换。

字段格式：

- `前置条件`、`操作步骤`、`预期结果` 必须使用 `1. 2. 3.` 换行分条。
- `操作步骤` 不得默认执行人已经位于当前模块页面，必须从系统或项目入口开始写完整导航路径，例如登录系统、进入一级菜单、进入二级/三级菜单或打开目标页面，再描述具体控件操作、输入、保存、提交、确认、取消、校验或查看结果。
- 示例：

```text
1. 测试环境已部署本次需求版本
2. 测试账号已完成注册且状态为启用
3. 用户已进入登录页
```

## Sheet 8：页面元素覆盖清单

适用场景：

- 输入包含页面截图、原型图、页面 URL、菜单路径、操作录屏或页面说明。
- 需要覆盖页面所有可点击/可交互功能。
- 文档不足时，需要通过浏览器或 computer use 观察页面后补齐元素清单。

页面实探数据安全：

- 对已有数据可以执行查看、搜索、筛选、排序、分页、打开详情，或进入编辑/修改页面观察字段、校验、联动、多级子菜单和保存前提示。
- 对既有数据必须主动做只读深探：可以进入详情、编辑页、删除/停用/提交确认弹窗观察字段、联动、二次确认、权限提示和取消路径，但不能最终确认。
- 可以复制既有数据字段，改名或改编码为带测试标识的新数据后新增，再用新数据探索详情页、下一级页面、编辑、删除、状态变更和失败分支。
- 不得对已有数据执行保存、提交、确认修改、删除、审批、驳回、启用、禁用、发布、下线、分配、转移、批量处理或状态变更。
- 新增、编辑、删除、状态变更、批量处理、外部调用等写操作只允许作用于本次实探自己创建且带有明显测试标识的数据，例如 `AI_TEST`、`CODEX_TEST`、当前日期或任务编号。
- 如果无法确认数据是否由自己创建，必须按已有数据处理。
- 对本次创建且带测试标识的数据，可以执行可恢复的新增、编辑、删除和配置验证；不可逆、影响权限、生产通知或外部系统的操作仍需用户明确授权和隔离环境。对已有数据只能观察，不得保存提交或改变状态。

定位：

- 本 Sheet 是页面元素覆盖追踪矩阵，不是测试用例 Sheet。
- 本 Sheet 只记录页面元素、业务依据、覆盖状态、发现方式、素材来源和关联的 `覆盖用例 ID`。
- 所有功能测试用例必须写入 `功能测试用例` Sheet。
- 所有性能测试场景必须写入 `性能测试设计` Sheet。
- 不得在本 Sheet 中编写独立测试用例、操作步骤、测试数据或完整预期结果正文。

字段：

- 元素 ID
- Story ID/需求 ID
- 页面/入口
- 页面 URL/菜单路径
- 元素名称/文案
- 元素类型
- 交互方式
- 适用DFX维度
- 适用DFX场景
- 前置状态/权限
- 预期行为
- 业务依据/规则来源
- 覆盖用例 ID
- 覆盖状态
- 发现方式
- 素材来源
- 待确认问题/备注

字段说明：

- `元素类型` 固定值建议：按钮、链接、菜单、Tab、图标按钮、下拉项、输入框、上传控件、开关、复选框、单选框、表格行操作、批量操作、分页、排序、筛选、弹窗按钮、抽屉按钮、快捷入口、其他。
- `交互方式` 固定值建议：点击、双击、悬停、输入、选择、拖拽、上传、下载、快捷键、其他。
- `选项取值/输入值` 用于记录下拉框、级联选择、单选框、复选框、树选择、枚举筛选等选择类控件已经逐项实际选择过的有限选项或输入值，不得只写“查看选项”。
- `联动/依赖变化` 用于记录每个有限选项触发的后续字段、二级/三级下拉、列表数据、按钮禁用态、校验提示、默认值、清空重置、搜索结果或提交前提示变化；无联动也要明确写“无联动变化”。
- `结果分支/后续状态` 用于记录输入或选择后实际进入的下一步、页面变化、结果列表、成功/失败/重复/为空/格式错误等分支、按钮状态、保存前校验和可恢复路径。
- `覆盖状态` 固定值：已覆盖、不适用、不测范围、待确认。
- `发现方式` 固定值建议：需求文档、截图、原型、浏览器实探、computer use、代码/DOM、用户说明。
- `业务依据/规则来源` 用于记录设计文档章节、需求说明、接口文档、验收标准、页面观察或待确认项，避免只按按钮机械生成用例。
- `覆盖用例 ID` 可填写一个或多个功能测试用例 ID；如果覆盖状态不是 `已覆盖`，必须在 `待确认问题/备注` 中说明原因。

覆盖规则：

- 所有可点击/可交互元素必须有覆盖状态。
- `已覆盖` 的元素必须关联功能测试用例。
- `覆盖用例 ID` 必须指向 `功能测试用例` Sheet 中的用例；性能相关元素或链路应通过 `性能测试设计` Sheet 中的性能场景 ID 体现。
- `不适用`、`不测范围`、`待确认` 的元素必须写明原因。
- 页面交互用例必须同时体现业务逻辑依据和具体页面操作。
- 不允许只覆盖页面主流程而遗漏行内按钮、图标按钮、二级菜单、更多操作、分页、排序、筛选、上传、下载、弹窗确认/取消等入口。

## 独立测试系统导入文件

适用场景：

- 需要将测试设计结果导入测试系统。
- 需要与测试系统导出的 `测试用例模板.xlsx` 字段顺序保持一致。

定位：

- 正式测试设计工作簿不再新增 `测试系统导入用例` Sheet。
- 导入内容必须写入 `docs/test-design/测试用例模板.xlsx` 的副本，不得修改原模板文件。
- 独立导入文件内容必须从 `功能测试用例` Sheet 映射生成。
- 所有功能测试用例仍以 `功能测试用例` Sheet 为准。
- 所有性能测试场景仍以 `性能测试设计` Sheet 为准。

字段顺序：

- 一级模块系统编号
- 一级模块名称
- 二级模块系统编号
- 二级模块名称
- 三级模块系统编号
- 三级模块名称
- 四级模块系统编号
- 四级模块名称
- 五级模块系统编号
- 五级模块名称
- 其他模块系统编号
- 其他模块名称
- 测试用例系统编号
- 测试用例序号
- 测试用例名称
- 测试步骤描述
- 测试步骤预期结果
- 测试类型
- 测试用例级别
- 执行方式
- 测试用例说明
- 前置条件
- 维护人
- 标签
- 备注
- 作者

映射规则：

- `测试用例名称` 来源于 `功能测试用例.用例标题`，必须保持正式、简洁、可检索，不得口语化或写成操作随笔；导入值必须保留 `功能点-当前用例标题` 格式，以弥补测试系统没有独立 `功能点` 字段的问题。
- 处理历史用例或外部输入时，如果 `功能测试用例.用例标题` 尚未带功能点前缀，生成独立导入文件前必须先按本行 `功能点` 字段规范化为 `功能点-当前用例标题`。
- `测试步骤描述` 来源于 `功能测试用例.操作步骤`。
- `测试步骤预期结果` 来源于 `功能测试用例.预期结果`。
- `前置条件` 来源于 `功能测试用例.前置条件`。
- `测试类型` 来源于 `功能测试用例.测试类型`，但必须转换为测试系统枚举值。
- `测试用例级别` 使用测试系统的 `L1/L2/L3/L4`，不得直接输出 `P0/P1/P2/P3`。
- `执行方式` 使用测试系统下拉枚举，只能填写 `自动化` 或 `手动`。
- `执行方式` 表示测试系统中该条用例当前如何执行，默认填写 `手动`。仅当存在已实现、可运行、可维护且覆盖该用例主要校验点的自动化脚本/API/流水线任务，并且本次交付明确要求按自动化执行导入或关联自动化资产时，才填写 `自动化`。
- `功能测试用例.是否适合自动化`、`自动化建议` Sheet、AI 页面实探或工具辅助点击结果，都不能直接映射为 `执行方式=自动化`。
- 模块名称列优先按业务模块层级拆分；没有系统编号时，模块系统编号列可以留空。
- `测试用例系统编号`、`作者` 等测试系统自动生成字段不需要填写，不得编造值。
- `标签` 可由模块、功能点、`DFX维度`、`DFX场景` 和风险类型组合生成，便于导入测试系统后检索 DFX 覆盖。
- `备注` 应保留 DFX 落地说明，例如 `DFX覆盖：DFS安全-权限控制`；不得只在正式测试设计中体现 DFX 而导入文件完全丢失。
- 测试系统模板中的红色字段和必填字段必须按每条测试用例动态生成，不得固定写示例值或通用占位值。
- 红色字段包括 `测试用例名称`、`测试步骤描述`、`测试步骤预期结果`、`测试类型`、`测试用例级别`、`执行方式`。
- 必填字段至少包括 `测试用例名称`、`测试类型`、`测试用例级别`、`执行方式`；如后续测试系统模板或字段解释发生变化，应以最新模板和字段解释为准。
- 带下拉框的字段只能填写测试系统模板允许值，不得自由发挥；当前下拉字段包括 `测试类型`、`测试用例级别`、`执行方式`。
- 输出独立导入 `.xlsx` 时必须保留模板中的 Excel 数据验证下拉框、必填字段样式、标红字段和自动生成字段空值；不得通过新建空白 Sheet 或仿制表头替代模板副本。生成程序只能填充内容，新增数据行的边框、字体、填充、对齐、数字格式和下拉验证范围必须沿用模板第 2 行示例数据格式。
- `测试用例序号` 应根据模块层级和小功能块顺序动态生成，支持层级化序号，例如 `1`、`1.1`、`1.2`。
- `测试用例说明`、`标签`、`备注`、模块名称等字段也应结合用例内容动态生成，不得整列重复无意义文本。

## 客户交付件与内部产品版图

测试资产事实必须保存在项目文件中，不依赖 AI 对话记忆。客户交付件与内部维护资产必须分离。

客户交付目录：

- `docs/test-design/current/`：保存当前任务交付件。
- `docs/test-design/deliverables/`：保存已交付给客户或测试系统的文件副本。

内部资产目录：

- `docs/test-assets/product-map.xlsx`：产品测试知识图谱主入口。
- `docs/test-assets/modules/`：保存正式测试设计最终版。
- `docs/test-assets/imports/`：保存测试系统导入文件副本。
- `docs/test-assets/indexes/`：保存兼容或补充索引。

`product-map.xlsx` 不作为默认客户交付件。客户交付件只包含本次任务范围，不默认包含内部产品版图、全量历史用例索引、全量业务链路或内部可复用测试数据。

`product-map.xlsx` 至少包含以下 Sheet：

- `产品模块地图`
- `业务对象地图`
- `业务链路地图`
- `页面元素地图`
- `用例资产索引`
- `模块能力索引`
- `跨模块依赖关系`
- `可复用测试数据`
- `变更影响分析`
- `变更记录`

生成或补充测试用例前，按需读取与当前功能相关的产品版图和历史归档。页面是已实现功能的主要事实来源，需求和模型推断只作理解参考。页面可验证的问题由模型继续实探；只有页面、文档和现有数据都无法确认且答案会实质改变设计时才询问用户。用户确认后只增量补探受影响分支，并将结果更新到 `风险与待确认问题` Sheet。

单模块任务正式写用例前，也要先做模块级粗遍历，识别菜单入口、页面清单、核心功能点、业务对象、状态流转和跨模块依赖。具体写测试用例时，要在对应页面或功能点内做深遍历，覆盖所有可点击、可输入、可测试元素。

深遍历下拉框、级联选择、单选框、复选框、树选择、枚举筛选等有限选择类控件时，不得只展开查看选项；必须逐项选择并观察前后依赖关系，每个选项规划独立 baseline Case。选择类控件标记为`已实测`时，`page-discovery.csv` 必须填写实际取值、联动/依赖变化和结果分支。

深遍历输入框、搜索框、文本域、数字框、日期框、URL/地址、端口、邮箱、手机号、名称、编码等输入类控件时，不得只观察字段存在或只写“输入后校验”；必须实际输入正常、异常、边界或用户提供的测试数据并触发后续动作。输入类控件已生成用例时，`page-discovery.csv` 必须填写 `选项取值/输入值`、`预期/观察行为` 和 `结果分支/后续状态`。

新增、创建、添加、新建、保存、提交、下一步、完成、测试连接等新增类流程必须实填实走，在安全边界内实际填写测试数据并触发后续动作。新增类流程已生成用例时，`page-discovery.csv` 必须填写提交数据、成功/失败真实结果、进入的下一级页面或失败后的停留页面；成功后进入的详情页、下一级页面或后续配置页也必须继续补充页面元素覆盖清单和功能测试用例。

本轮事实必须沉淀到 run-dir 的 `page-discovery.csv`。`product-map.xlsx` 只在用户需要历史资产复用、跨模块依赖或长期产品版图时按需同步，不参与本轮完成判断。

生成后，统一工具将正式测试设计与导入文件原子发布到 `current/`、`deliverables/` 和对应内部归档。仅在本轮启用产品版图时同步相关历史资产。

生成正式测试设计 Excel 后，应运行统一交付校验。`page-discovery.csv` 用于核对事实与用例；`product-map.xlsx` 仅在显式传入时同步和校验。`batch-status.csv` 只做结构、数量和断点一致性检查，不作为语义门禁。

测试系统导入文件必须优先使用 `scripts/test_design_excel_tools.py compile-deliverables` 从现有JSON分片一站式生成；如只需单独生成导入文件，可使用 `generate-import`。禁止在批次目录创建任务专用Python或按固定列序号数组直接写入模板。导入文件生成后必须通过交付件校验确认字段未错位、自动生成字段为空、下拉字段合法、模板数据验证保留、多行字段开启自动换行。

正式测试设计和导入文件中的多行字段必须设置自动换行和顶部对齐，不得只依赖模板前几行样式继承。正式测试设计至少包括 `前置条件`、`测试数据`、`操作步骤`、`预期结果`、`备注`；导入文件至少包括 `测试步骤描述`、`测试步骤预期结果`、`前置条件`、`测试用例说明`、`备注`。新增数据行必须复制模板第 2 行示例数据格式，只填充内容，不得改变边框、字体、填充、对齐、数字格式、下拉验证和必填/标红样式。

正式测试设计、导入文件和产品版图不得保留 Excel Table 对象或 `/xl/tables/table*.xml` 部件；页面元素覆盖清单等 Sheet 使用普通单元格区域、样式和自动筛选。不得出现打开文件时 Microsoft Excel 提示修复、表格对象仍停留在模板前三行、自动筛选范围未覆盖新增行等问题。使用 `scripts/test_design_excel_tools.py generate-import` 或 `fix-formal-styles` 时，工具必须移除 Excel Table 对象并刷新自动筛选范围。

新批次交付使用 `scripts/test_design_excel_tools.py compile-deliverables` 一次完成分片解析、8-Sheet编译、导入文件生成、双文件校验和原子发布，并在成功后回写`batch-status.csv`的实际数量、状态和路径。`complete-deliverables`仅用于历史正式草稿恢复。传入`--page-discovery`时必须同时传入`--batch-status`；仅在需要长期资产同步时显式传入`--product-map`。

交付文件命名必须以菜单/模块路径为准，使用 `一级模块_二级菜单_三级菜单[_四级]_测试设计.xlsx` 和 `一级模块_二级菜单_三级菜单[_四级]_导入用例.xlsx`；不得把运行文件夹名、批次目录名或产品名拼入交付文件名。产品名应写入产品版图和工作簿字段，不作为 `current/`、`deliverables/`、`docs/test-assets/modules/`、`docs/test-assets/imports/` 中文件名的一部分；如 `module-path` 中包含产品名前缀，应同时传入 `--product-name` 让统一工具自动去除。

交付件不得残留 `{NAV}`、`{NL}`、`{Q}`、`{E}`、`${...}`、`{{...}}`、`TODO`、`TBD` 等模板占位符或未完成标记。当前批次各 Sheet 的内容必须与最小标题路径一致，不得残留其他模块的模板内容、示例内容或无效用例 ID 引用。

跨模块用例只保留一个主归属模块。当前模块依赖已有模块能力时，应优先引用已有用例 ID 作为前置条件，不重复复制已有用例。


保留现有按功能块拆分的多个JSON分片，避免一次写入超大正文。每个分片使用8个正式Sheet标题作为顶层section，section内使用`rows`数组；本功能的故事、场景、用例、性能、风险和自动化建议写入同一分片，测试设计总览只在一个分片写入。内部只调用仓库标准CLI，不在run-dir生成任何Python；问题直接修改所属事实或源分片。

页面实探运行保留`batch-status.csv`、`page-discovery.csv`和`artifacts/`；首次写入事实时由工具自动创建run-dir，不向用户暴露初始化阶段。多页面、多模块或大范围任务额外保留精简`batch-plan.md`；单页面不强制计划，也不预创建空白Review。`batch-status.csv`只承担进度、数量、交付路径和断点恢复，不作为语义门禁。

可选诊断截图或过程材料只能写入当前任务的 `docs/test-assets/batch-runs/<task>/artifacts/`，不得写入共享根目录；证据不参与完成判断、Case规划、Review或Excel生成。

`batch-status.csv` 和 `page-discovery.csv` 必须从 `docs/test-assets/batch-runs/templates/` 复制标准模板或按完全相同表头生成，禁止自定义精简表头。写入 `page-discovery.csv` 时必须使用 CSV writer 或等价结构化写入方式，保证每一行列数与表头一致，防止字段错位；不得手工拼接逗号分隔字符串。

每个批次正式写测试用例前，如果存在可访问页面、原型或桌面窗口，必须使用浏览器能力或 computer use 遍历当前批次最小标题路径下所有可点击/可交互功能点，并把发现结果写入`page-discovery.csv`；页面元素覆盖清单由统一编译器生成。只有显式启用长期资产同步时才更新产品版图。

每次写入事实或JSON分片时执行轻量生产者自检。最终交付时核对`page-discovery.csv`、正式Excel、导入文件和模块归档；仅在显式启用产品版图时同步`product-map.xlsx`。

DFX在Case规划阶段按功能适用性驱动扩展，不机械展开 12×4 矩阵。大范围任务按可连续完成的最小功能范围分批，完成一个分片的轻量结构自检后即可继续；全部产物完成后只做一次跨产物语义Review，不设置逐批语义拒绝器或自动返工循环。
首次交付后的补充、追加、二次补充或页面未覆盖反馈必须走增量补充流程，不得只追加用例。生成补充 Excel 前，必须读取产品版图、归档测试设计、现有交付件、页面元素覆盖清单、`page-discovery.csv` 和 `batch-status.csv`，识别覆盖缺口、受影响最小标题路径、已有用例 ID 和可复用历史用例；随后建立或更新补充批次，并按最小标题路径重新页面实探目标覆盖缺口。二次补充输出必须合并到正式测试设计和独立导入文件副本，新增用例放在对应小功能块附近，能复用已有用例时引用已有用例 ID，不重复复制，并同步页面元素覆盖清单、性能测试设计、风险与待确认问题、自动化建议、`docs/test-assets/modules/`、`docs/test-assets/imports/` 和 `product-map.xlsx`。

启用`product-map.xlsx`同步时，写入的当前功能资产不得残留模板示例行；未启用时不创建、不强制填写，也不影响双Excel交付。

正式交付件、导入文件、批次账本和产品版图允许保留本轮内网测试地址、账号、凭据和载荷，不执行敏感内容识别、脱敏、替换或交付拦截。
交互按操作类型判断闭环：选择、输入、查询以具体观察结果为闭环；新增、编辑、删除和配置变更必须验证保存、回显、生效、取消或删除结果；不要求所有临时交互追加通用恢复语句。
测试用例必须详细覆盖每个功能自身的baseline Case和适用DFX方向；不适用方向不机械展开，不默认生成组合矩阵，也不能用一个笼统Case替代多个独立功能。

枚举建议：

- `测试类型`：功能测试、性能规格测试、可靠性测试、兼容性测试、可维护性测试、安全性测试、易用性测试。
- `测试用例级别`：L1、L2、L3、L4。
- `执行方式`：手动、自动化。无已落地自动化资产时必须填写 `手动`。

## 样式和可用性要求

- 所有 Sheet 表头冻结。
- 所有 Sheet 表头加粗并使用统一底色。
- `前置条件`、`操作步骤`、`预期结果`、`测试数据`、`备注`、`预期行为`、`业务依据/规则来源`、`待确认问题/备注` 等长文本列启用自动换行。
- 表格开启筛选。
- 枚举字段应使用固定值。
- 工作簿内保留示例行，便于用户理解填写方式。

## 外网到内网升级

普通框架升级使用升级包同步规范、模板和脚本，不覆盖内网业务资产。受保护目录包括：

- `docs/test-assets/`
- `docs/test-design/current/`
- `docs/test-design/deliverables/`

标识：PROTECTED_ASSET_DIRS

`VERSION` 中的 `framework_version` 表示框架版本，`asset_schema_version` 表示内部资产结构版本。`product-map.xlsx` 是主要可能演进的数据结构；历史归档 Excel 默认作为历史快照保留。如果 `asset_schema_version` 或产品版图结构变化，必须通过迁移脚本读取旧资产并增量补齐，不得用空模板覆盖真实资产。
- 分批必须按当前产品或模块可识别的最深标题级别执行，例如一级标题、二级标题、三级标题、四级标题等，哪个标题级别最小就以哪个最小标题作为一个批次。每个已通过批次只能覆盖 1 个最小标题路径，`最小标题路径` 使用 `一级>二级>三级>四级` 形式记录唯一叶子节点；禁止合并多个最小标题，禁止再拆分一个最小标题为多个批次。已通过批次的导入文件路径必须真实存在，并能与归档测试设计逐个匹配校验。
- 当任务范围是全产品、多个一级模块或大模块时，必须先遍历一级菜单、二级菜单、三级菜单及更深层标题，拿到菜单轮廓、页面清单和功能地图后输出分批设计计划，不得一次性生成完整测试用例；各批按最小标题路径执行，所有批次完成后再做跨模块汇总。
- 当任务范围超过一个最小标题时，必须建立批次队列并按最小标题路径逐批执行。
