# 测试计划与用例规则

## 独立功能与计划

从 facts 识别具有独立触发和可观察结果的用户功能，例如搜索、重置、排序、分页、详情、新建、编辑、删除和导出。控件、分页条数和普通字段不是独立功能。

每个功能先建立基线意图，再根据事实适用性展开 DFX。一个事务的多个检查点可以合并成一条用例，也可以按不同可观察结果拆分；不得按点击数固定生成用例。每个事务检查点必须分配给某条 Case，或明确进入 performance、risk、not_applicable。

## 用例正文

- 只按 `case-plan.json` 的稳定 Case ID 和功能顺序写作，不临时发明功能、场景和预期。
- 标题使用“功能点-具体场景”，不得为空。
- 每条用例第一步使用完整实际菜单路径，例如“进入告警管理-告警列表”，不逐级拆分菜单。
- 前置条件只描述角色、环境、状态和数据；菜单导航写入步骤。
- 使用具体且脱敏的数据，不写“输入测试数据”等占位描述。
- 每个步骤使用 `{action, expected}` 配对；预期必须可观察、可判定且来源于实探事实。
- 同功能用例连续排列。公共导航允许重复，核心操作、数据和预期不得整段重复。
- 禁止截图要求、UID、DOM、选择器、事实编号和页面工具术语。

CRUD、编辑和配置用例必须写出提交后的查询、重开回显、实际效果和恢复/清理。无效分支写清具体拦截与数据未变化，不能编造成成功结果。
