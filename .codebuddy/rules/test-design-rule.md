# 测试设计硬规则

1. 单会话、单浏览器上下文顺序执行；不得创建或调用多 Agent，不得依赖页面操作 Hook。
2. 用户直接调用 Skill；运行目录自动绑定或恢复，不出现独立初始化阶段。
3. 默认全量深探并开放发现未知控件；扫描和事务交替，不建立逐元素义务队列。
4. 元素登记先归一化常见页面模型类型、对象/字符串形式的输入类和选项；非空 `options` 默认是有限选项，未知交互类型或未说明来源的空选择控件不得静默跳过。再根据页面语义、需求参考和模型推断声明各 `valid_input_classes`，由 DFX 策略连同必填、格式、边界、有限选项和配置基线生成 `exploration_requirements`；每个有效输入类、每个有限选项分别实测，必填输入同时实测空值，明确格式/边界时实测相应分支；需要按钮触发时检查包含完整触发动作；CRUD、普通编辑和单因素配置验证提交、持久化回显、实际效果和恢复/清理。
5. 页面可验证的问题必须自行操作；只有外部业务语义、权限、数据或环境阻塞进入 `open_items`。
6. 证据是可选诊断，不是事实、计划、Review 或交付的输入，不默认创建证据目录。
7. 独立功能先有基线用例；每个有限选项和每个实测有效输入等价类分别形成独立 baseline Case，已声明并实测的空值、格式、边界等分支分别形成独立 DFX Case；不合并、不默认做笛卡尔积，也不使用固定矩阵或固定用例数。
8. 一个用例归属一个功能；辅助使用其他控件不能替代该控件所属功能的独立覆盖。
9. 已完成的实探检查立即写入；`element_ref` 表示主验证，`used_element_refs` 表示全部参与控件，辅助使用不得完成其自身独立分支，独立分支不得复用同一物理动作冒充。检查记录适用的 `input_class`、`option_value`、稳定结果锚点、已观察动态状态和非预期结果；非预期结果必须进入用例、风险或性能结论。未执行分支保留在既定清单中，不拒绝已完成事实，也不在 checkpoint 临时新增义务；事务结构错误仍在写入现场校验。调用方误放在 `data` 内的事件保留字段在落盘前统一提升，值冲突一次性拒绝，不能先产生幽灵事实再靠checkpoint清理。最后一次事务后必须再做一次稳定全量扫描并明确未处理元素为空；相同扫描幂等吸收，新事务后旧扫描自动失效。facts按页面checkpoint刷新、保持首次发现顺序且可自动重建。计划只维护唯一 `check_assignments`。
10. 每条计划Case必须有唯一 `verification_focus`，说明主验证对象、独立分支和可观察效果；运行时可从实探事实派生建议，并为默认有限选项提供“切换后再切回”的动作意图以区别输入类Case。模型只写业务action+expected，运行时根据页面事实统一注入使用 `-` 连接的完整菜单导航；动作保留实测选项/输入和触发，预期围绕主验证目标并优先使用稳定tokens，不固化偶发计数、百分比、耗时或进度。系统自动注入内部来源和主验证目标且不导出内部引用。计划每功能一次写紧凑设计上下文和自动化画像；用例只写自动化价值、优先级和必要覆盖。
11. 禁止把截图、UID、DOM、选择器、内部编号、工具操作和占位数据写入可执行用例；命名受控数据使用 `TEST_*` 引用并在前置条件说明同名来源。
12. 同功能用例连续排列；公共导航可以重复，核心操作、数据和预期不得整段重复。
13. Review 只做一次跨产物审计；运行时重新计算 discovery、plan、cases 和跨产物确定性结果，模型在同一负载中标明已审 `cases/performance/risks/automation/elements/cross_sheet`，集中检查主验证目标差异、步骤预期对应、专项依据、风险去重、自动化入口、元素分支完整性和跨Sheet一致性，只提交实际语义结论和问题，不提交自报通过清单。缺少语义审计不得交付。Review不新增DFX场景，不触发全量返工或自动重试，只允许一次集中局部修正。
14. 正式测试设计与测试系统导入 Excel 从各自模板、同一结构化用例源独立生成；8个Sheet只组合上游设计上下文、计划、风险、性能、自动化和事实，不自行发明业务默认内容；元素覆盖使用简短中文完整列出全部分支和实际交互，不输出“另有N项”、内部输入分类词或通用trigger占位语；风险编号和等级在总览、场景矩阵、功能用例及风险Sheet中来自同一映射。不得出现空标题、中间空行、跨Sheet字段不一致或文本截断。
15. 既有数据只读；只修改或删除本轮带 `AI_TEST`、`CODEX_TEST` 或用户明确标识的测试对象。
16. 保护 `docs/test-assets/`、`docs/test-design/current/`、`docs/test-design/deliverables/` 中已有用户资产。
17. 内部执行只使用标准CLI和JSON负载，禁止临时Python编排脚本；新运行目录固定为 `docs/test-design/current/<run-id>/`，已存在的历史运行可原地恢复，运行目录不得逃逸当前项目根目录。
18. `case-plan.json` 与 `function-cases.json` 在各自单文件内按功能幂等 upsert；精确重复提交直接吸收，不建立分片、模糊去重或自动重试状态机。
19. 计划必须明确性能与风险是否适用；按钮、提交动作或 `trigger_element_ref` 本身不是性能证据，只有实测加载、异步、长任务、超时、耗时或需求性能目标时才派生一条轻量响应场景，量化目标必须带需求或实测事实引用。自动化画像中同类稳定性风险聚合为一个风险及受影响功能，页面用例自动化入口归一为UI；其余专项内容由模型按事实补充，不按Case重复。确无适用内容时保留真实不适用原因。恢复时只读取固化产物；负载JSON使用系统临时目录并清理。命令不可用时只允许降级到同一 `execute_request` 入口一次，不直接调用底层函数。双Excel存在且Review有效时状态为完成，不重复生成。
