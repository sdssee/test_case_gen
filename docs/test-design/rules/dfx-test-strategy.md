# DFX 左移测试策略

DFX 是独立功能基线上的扩展策略，不是替代页面元素覆盖的用例生成器。它有两次左移但只有一个策略来源：元素登记时，模型结合页面语义、功能推断和已提供需求参考声明有效输入类，运行时再把它们与必填、边界和有限选项编译为 `exploration_requirements`，供深探交互前执行；`case-plan.json` 只根据已观察结果形成基线与适用DFX意图，不重复发现页面分支。不能等事务或用例写完后由门禁临时新增义务再补。

## 适用维度

| 维度 | 典型扩展 |
| --- | --- |
| DFT 功能 | 正向、边界、异常输入、取消/逆向 |
| DFB 业务 | 业务流程、数据准确、端到端、统计 |
| DFS 安全 | 权限、越权、脱敏、输入防护 |
| DFR 可靠 | 持久化、一致性、幂等、恢复 |
| DFU 可用 | 反馈、提示、禁用状态、引导 |
| DFI 接口 | 参数、错误码、超时与协议（有接口证据时） |
| DFC 兼容 | 浏览器、屏幕、格式（有明确范围时） |
| DFP 性能 | 响应、并发、大数据量、资源监控 |
| DFM/DFD/DFO | 维护、部署、运维专项 |
| DFX 极端 | 压力、耗尽、破坏性专项 |

每个实测有效输入等价类和有限选项先形成独立 DFT baseline Case；空值、无效等价类和边界值属于 DFT 扩展。权限、状态一致性、持久化等按相应维度扩展。两个独立维度默认不做笛卡尔积，仅选择与事实和需求相关的场景，不按矩阵机械生成固定数量。

DFP 和破坏性极端场景默认进入“性能测试设计”或风险，不伪装成功能用例。按钮、提交动作和触发控件引用不等同于性能事实；存在实测加载、异步、长任务、超时、耗时或明确需求目标但没有并发容量目标时，只形成一条共享的轻量响应与超时场景，未提供量化目标时不得发明秒数。无可观察性能行为时判定不适用。页面可以直接验证的权限、提示、一致性和恢复场景可以进入功能用例。

每个计划用例必须说明 `strategy`、`dfx_dimension`、`dfx_scenario` 和引用事实；基线用例的 `strategy=baseline`，DFX 扩展不得消灭独立功能的基线覆盖。
